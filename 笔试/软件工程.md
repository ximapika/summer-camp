# 软件工程

软件工程是用系统化、规范化的方法来开发和维护软件的学科。它的目标是在预算和时间限制内，交付高质量的软件产品。软件工程关注的不仅是编写代码，还包括需求分析、系统设计、项目管理、测试、维护等软件生命周期的各个阶段。

## 一、软件过程模型

软件过程模型（也称软件开发生命周期模型）定义了软件开发的各个阶段及其顺序。选择合适的过程模型对项目的成功至关重要——没有万能的模型，选择取决于项目特点（需求是否明确、规模大小、风险程度、团队能力等）。

| 模型 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|----------|------|------|----------|
| **瀑布模型** | 线性顺序：需求→设计→编码→测试→维护，每个阶段完成后才进入下一个 | 简单直观，文档完整，阶段清晰 | 不适应需求变化，后期发现问题修复代价高，用户要到最后才看到产品 | 需求明确且稳定的项目（如嵌入式系统） |
| **增量模型** | 将系统分为多个增量，每次开发和交付一个增量 | 核心功能先交付，用户尽早使用，降低风险 | 需要开放的架构支持增量集成 | 需要快速交付核心功能 |
| **螺旋模型** | 每轮迭代包含：目标确定→风险分析→开发验证→计划，螺旋式推进 | 重视风险管理，适合大型项目 | 需要丰富的风险评估经验，成本高 | 大型、高风险、需求不完全明确的项目 |
| **原型模型** | 快速构建原型供用户评估，根据反馈迭代改进 | 有助于澄清需求，用户早期参与 | 原型可能被误认为成品，代码质量可能不高 | 需求不明确、用户难以描述需求 |
| **敏捷开发** | 以人为核心，迭代增量开发，拥抱变化，快速反馈 | 响应变化能力强，持续交付价值 | 文档较少，对团队协作能力要求高 | 需求多变、团队小而精 |

### 1.1 敏捷方法

**敏捷宣言**（2001 年）提出了四条核心价值观：
- **个体和互动** 高于 流程和工具
- **工作的软件** 高于 详尽的文档
- **客户合作** 高于 合同谈判
- **响应变化** 高于 遵循计划

**Scrum** 是最流行的敏捷框架：

- **三个角色**：
  - **Product Owner（产品负责人）**：定义产品需求和优先级，维护 Product Backlog
  - **Scrum Master（敏捷教练）**：确保团队遵循 Scrum 流程，移除障碍
  - **开发团队**：自组织的跨职能团队，通常 5-9 人

- **四个活动**：
  - **Sprint Planning（冲刺计划）**：确定本轮 Sprint（通常 2-4 周）要完成的用户故事
  - **Daily Standup（每日站会）**：15 分钟内每人回答三个问题——昨天做了什么、今天要做什么、有什么障碍
  - **Sprint Review（冲刺评审）**：向利益相关者演示本轮成果，收集反馈
  - **Sprint Retrospective（冲刺回顾）**：团队反思本轮做得好/不好的地方，持续改进

- **三个工件**：
  - **Product Backlog**：按优先级排序的所有需求列表
  - **Sprint Backlog**：本轮 Sprint 选中的需求子集及实现计划
  - **增量 (Increment)**：本轮 Sprint 产出的可交付软件

**XP (极限编程)**：
- **结对编程**：两人共用一台电脑，一人写代码（驾驶员），一人审查（领航员），定期交换角色
- **TDD (测试驱动开发)**：先写测试用例，再写代码使测试通过，最后重构。红-绿-重构循环
- **持续集成 (CI)**：开发人员频繁（每天多次）将代码集成到主干，每次集成自动构建和测试
- **重构**：不改变外部行为的前提下改善代码内部结构
- **小发布**：频繁发布小版本，快速获取用户反馈

**看板 (Kanban)**：
- 可视化工作流（通常用看板板：To Do → In Progress → Done）
- **限制在制品 (WIP)**：控制同时进行的工作数量，避免过载
- 持续流动，不需要固定周期的 Sprint

## 二、需求工程

需求工程是软件开发的第一步，也是最容易出错的一步。据统计，约 40%-60% 的软件缺陷可追溯到需求阶段的问题。需求工程的目标是准确、完整地理解和记录用户需要什么。

### 2.1 需求分类

- **功能需求**：系统必须提供的功能和服务。例如"用户可以通过邮箱注册账号"、"系统支持按关键词搜索商品"
- **非功能需求**：对系统的质量约束。包括：
  - 性能：响应时间 < 200ms、支持 10000 并发
  - 可靠性：系统可用性 99.9%
  - 安全性：数据加密传输、防止 SQL 注入
  - 可维护性：代码可读性、模块化
  - 可用性：界面友好、操作步骤少
  - 可移植性：支持 Windows/Linux/macOS

### 2.2 需求工程过程

1. **需求获取 (Elicitation)**：通过各种方式了解用户需要什么
   - 技术：用户访谈、问卷调查、观察用户工作、分析竞品、头脑风暴、用例分析
   - 挑战：用户往往说不清楚自己要什么，不同利益相关者需求可能冲突

2. **需求分析 (Analysis)**：对收集到的需求进行整理和建模
   - 建模工具：用例图（描述功能和参与者）、数据流图（描述数据处理过程）、状态图
   - 确定需求的优先级（必须有 / 应该有 / 可以有）
   - 识别需求之间的冲突和依赖

3. **需求规格说明 (Specification)**：将分析结果形式化为文档
   - 撰写 **SRS (Software Requirements Specification，软件需求规格说明书)**
   - 需求应当是：明确的、可测试的、可追踪的、一致的

4. **需求验证 (Validation)**：确认需求的正确性和完整性
   - 评审会议、原型验证、需求走查
   - 验证需求是否准确反映了用户的真实意图

## 三、软件设计

### 3.1 设计原则 (SOLID)

SOLID 是面向对象设计的五条基本原则，由 Robert C. Martin（"Uncle Bob"）总结。遵循这些原则可以使代码更易于理解、扩展和维护。

| 原则 | 全称 | 含义 | 违反的后果 |
|------|------|------|-----------|
| **S** | Single Responsibility Principle（单一职责） | 一个类应该只有一个引起它变化的原因。每个类只负责一件事 | 类过于庞大，修改一个功能可能影响其他功能 |
| **O** | Open-Closed Principle（开闭原则） | 软件实体应对扩展开放，对修改关闭。通过增加新代码来增加新功能，而不是修改已有代码 | 每次新增功能都要修改现有代码，引入新 bug 的风险高 |
| **L** | Liskov Substitution Principle（里氏替换） | 子类必须能替代父类使用，而不破坏程序的正确性。子类不能削弱父类的行为 | 使用继承后出现意外行为（经典反例：正方形 is-a 矩形？） |
| **I** | Interface Segregation Principle（接口隔离） | 客户端不应被迫依赖它不使用的接口。接口应该小而专一 | 实现类被迫实现一堆用不到的方法 |
| **D** | Dependency Inversion Principle（依赖反转） | 高层模块不应依赖低层模块，二者都应依赖抽象。抽象不应依赖细节 | 高层逻辑被底层实现绑定，难以替换和测试 |

**其他重要设计原则**：

- **DRY (Don't Repeat Yourself)**：每一项知识在系统中应当有一个唯一、明确的表示。避免代码重复——重复的代码意味着修改时需要改多处，容易遗漏
- **KISS (Keep It Simple, Stupid)**：保持简单。简单的设计更容易理解、测试和维护
- **YAGNI (You Aren't Gonna Need It)**：不要实现当前不需要的功能。过度设计会增加复杂性和维护成本
- **高内聚低耦合**：
  - **内聚 (Cohesion)**：模块内部各元素的关联程度。高内聚意味着模块内的功能紧密相关
  - **耦合 (Coupling)**：模块之间的依赖程度。低耦合意味着修改一个模块不会影响其他模块
- **组合优于继承**：优先使用对象组合而非类继承来实现代码复用。继承会带来强耦合

### 3.2 架构模式

架构模式是软件系统整体结构的模板，决定了系统的组件划分和组件间的交互方式。

| 模式 | 描述 | 优点 | 缺点 | 典型应用 |
|------|------|------|------|----------|
| **分层架构** | 系统分为若干层，每层只能调用下一层的服务。典型：表示层→业务逻辑层→数据访问层 | 职责清晰，易于理解和维护 | 层间可能有不必要的数据传递 | 传统 Web 应用（MVC 框架） |
| **MVC** | Model（数据和业务逻辑）、View（用户界面）、Controller（处理用户输入，协调 M 和 V）三者分离 | 关注点分离，支持多种视图 | 控制器可能过于复杂 | Web 框架（Django, Spring MVC） |
| **微服务** | 将应用拆分为一组小的、独立部署的服务，每个服务实现一个业务功能 | 独立开发/部署/扩展，技术栈灵活 | 分布式系统复杂度（网络、一致性） | 大规模互联网系统（Netflix, Amazon） |
| **事件驱动** | 组件通过发布/订阅事件进行异步通信 | 松耦合，可扩展 | 调试困难，事件流难追踪 | 消息队列系统（Kafka） |
| **管道-过滤器** | 数据流经一系列处理步骤（过滤器），每步的输出是下一步的输入 | 各步骤独立，可复用 | 不适合交互式应用 | Unix 命令管道、数据处理 ETL |

### 3.3 UML 图

UML (Unified Modeling Language) 是一种标准化的建模语言，用图形化方式描述软件系统的结构和行为。UML 图分为结构图（静态）和行为图（动态）两大类。

| 图 | 类型 | 用途 | 关键元素 |
|----|------|------|----------|
| **用例图** | 行为图 | 描述系统提供的功能和参与者（Actor）之间的关系 | 参与者、用例、关系（包含/扩展/泛化） |
| **类图** | 结构图 | 描述系统中的类及其属性、方法、关系 | 类、接口、关系（继承/实现/关联/依赖/聚合/组合） |
| **时序图** | 行为图 | 描述对象之间随时间的交互顺序 | 生命线、消息（同步/异步）、激活框 |
| **状态图** | 行为图 | 描述一个对象在其生命周期内的状态转换 | 状态、转换、事件、守卫条件 |
| **活动图** | 行为图 | 描述工作流程或算法的步骤 | 活动、决策节点、分叉/汇合、泳道 |
| **组件图** | 结构图 | 描述系统组件及其依赖关系 | 组件、接口、依赖 |
| **部署图** | 结构图 | 描述系统的物理部署（节点和组件的对应关系） | 节点、部署的组件 |

**类图中的六种关系**（从弱到强）：

| 关系 | 图示 | 含义 | 示例 |
|------|------|------|------|
| **依赖 (Dependency)** | 虚线箭头 | 一个类使用了另一个类（最弱的关系） | 方法参数、局部变量 |
| **关联 (Association)** | 实线箭头 | 一个类持有另一个类的引用 | Student 有 Course 列表 |
| **聚合 (Aggregation)** | 空心菱形 | 整体-部分关系，部分**可以**独立存在 | 班级和学生（学生可脱离班级） |
| **组合 (Composition)** | 实心菱形 | 整体-部分关系，部分**不能**独立存在 | 人和心脏（心脏不能脱离人） |
| **实现 (Realization)** | 虚线空三角 | 类实现接口 | ArrayList 实现 List 接口 |
| **泛化 (Generalization)** | 实线空三角 | 继承关系 | Cat 继承 Animal |

## 四、软件测试

软件测试的目的是发现软件中的缺陷（bug），验证软件是否满足需求。Dijkstra 说过："测试只能证明 bug 的存在，不能证明 bug 的不存在。"尽管如此，充分的测试仍然是保证软件质量的最重要手段。

### 4.1 测试级别

软件测试按照粒度从小到大分为四个级别，形成测试金字塔：

| 级别 | 测试对象 | 执行者 | 目标 | 数量 |
|------|----------|--------|------|------|
| **单元测试** | 单个函数、方法或类 | 开发者 | 验证最小代码单元的正确性 | 最多（金字塔底部） |
| **集成测试** | 模块之间的接口和交互 | 开发者/测试 | 验证模块组合后是否正常协作 | 较多 |
| **系统测试** | 整个系统 | 测试团队 | 验证系统整体功能和非功能需求 | 较少 |
| **验收测试** | 用户需求 | 用户/客户 | 确认系统满足业务需求，决定是否接受 | 最少（金字塔顶部） |

**集成测试策略**：
- **大爆炸集成**：所有模块一起集成测试。简单但定位问题困难
- **自顶向下集成**：从主模块开始，逐步集成下层模块（用桩 stub 代替未完成的模块）
- **自底向上集成**：从底层模块开始，逐步向上集成（用驱动 driver 调用底层模块）
- **三明治集成**：上层自顶向下，下层自底向上，在中间汇合

### 4.2 白盒测试 vs 黑盒测试

**白盒测试**（基于代码结构，又称结构测试/玻璃盒测试）：测试者了解代码内部逻辑，设计测试用例以覆盖代码的各种执行路径。

覆盖标准从弱到强：

| 覆盖标准 | 要求 | 强度 |
|----------|------|------|
| **语句覆盖** | 每条可执行语句至少执行一次 | 最弱——即使所有语句都执行过，可能仍有分支没测到 |
| **分支覆盖（判定覆盖）** | 每个分支（if 的 true/false）至少执行一次 | 较强——但复合条件（如 `if (A && B)`）内部可能未充分测试 |
| **条件覆盖** | 每个布尔子条件的 true/false 至少取一次 | 更强——`A && B` 中 A 和 B 都要分别取 true 和 false |
| **条件-判定覆盖** | 同时满足分支覆盖和条件覆盖 | 强 |
| **路径覆盖** | 覆盖所有可能的执行路径 | 最强——通常不可行（循环会导致路径数爆炸） |

**黑盒测试**（基于需求规格，又称功能测试）：不关心代码内部实现，只根据输入输出规格来设计测试用例。

| 方法 | 思路 | 示例 |
|------|------|------|
| **等价类划分** | 将输入分为若干等价类（同类中的任何值测试效果相同），从每类中选取一个代表值 | 输入年龄：有效类 [1,150]，无效类 [-∞,0] 和 [151,+∞] |
| **边界值分析** | 在等价类的边界上取值测试，因为 bug 最容易出现在边界 | 年龄边界：0, 1, 150, 151 |
| **因果图** | 分析输入条件之间的因果关系和约束，生成判定表 | 多条件组合的输入场景 |
| **错误推测** | 基于经验和直觉猜测可能出错的地方 | 空输入、特殊字符、极大值、并发 |

### 4.3 其他测试类型

- **回归测试 (Regression Testing)**：修改代码后，重新运行已有的测试用例，确保修改没有引入新 bug 或破坏原有功能。这是持续集成的核心——每次代码提交都自动跑一遍测试
- **冒烟测试 (Smoke Testing)**：对新构建的版本做快速检查，验证核心功能是否正常（"系统能不能冒烟启动"）
- **性能测试**：
  - 负载测试：在预期负载下测试系统表现
  - 压力测试：超出预期负载，找出系统瓶颈和崩溃点
  - 稳定性测试：长时间运行观察是否有内存泄漏等问题
- **安全测试**：渗透测试（模拟攻击者）、漏洞扫描、代码审计
- **A/B 测试**：在生产环境中将用户随机分为两组，分别使用不同版本，比较指标差异

## 五、版本控制与 Git 工作流

### 5.1 核心概念

Git 是分布式版本控制系统（详见开发工具篇）。这里重点介绍 Git 在软件工程中的工作流模型。

```
工作区 (Working Directory)  ←git add→  暂存区 (Staging Area)  ←git commit→  本地仓库  ←git push→  远程仓库
```

### 5.2 Git 工作流模型

| 工作流 | 特点 | 适用场景 |
|--------|------|----------|
| **Git Flow** | 分支严格：master（生产）、develop（开发）、feature/*（功能）、release/*（发布）、hotfix/*（热修复） | 版本发布周期明确的项目 |
| **GitHub Flow** | 简单：main 分支 + 短生命周期的 feature 分支 + Pull Request | 持续部署的 Web 应用 |
| **Trunk-Based** | 所有人直接在主干开发，分支存活不超过一天 | 高频发布、CI/CD 成熟的团队 |

**Git Flow** 的核心流程：
1. 从 develop 拉出 feature 分支进行功能开发
2. 功能完成后合并回 develop
3. 准备发布时从 develop 拉出 release 分支进行测试和修复
4. 发布后合并到 master 和 develop，在 master 上打 tag
5. 生产环境紧急 bug 从 master 拉出 hotfix 分支修复

**Pull Request (PR) / Merge Request (MR)**：在合并分支前请求代码审查的机制。提交者描述改动，审查者检查代码质量、逻辑正确性和风格一致性。Code Review 是保证代码质量的重要实践。

## 六、设计模式

设计模式是前人在面向对象软件设计中总结出的**可复用的解决方案模板**。由 GoF（Gang of Four，四人帮）在 1994 年的《设计模式》一书中系统总结了 23 种经典模式，分为创建型、结构型、行为型三大类。

### 6.1 创建型模式

创建型模式关注对象的创建过程，目标是将对象的创建与使用分离，使系统不依赖于对象如何被创建的具体细节。

| 模式 | 目的 | 核心思想 | 典型场景 |
|------|------|----------|----------|
| **单例 (Singleton)** | 确保一个类全局只有一个实例，并提供全局访问点 | 私有构造函数 + 静态方法获取实例 | 配置管理器、线程池、数据库连接池、日志记录器 |
| **工厂方法 (Factory Method)** | 定义创建对象的接口，让子类决定实例化哪个具体类 | 将 new 操作延迟到子类 | 框架中创建对象（如 Spring Bean 工厂） |
| **抽象工厂 (Abstract Factory)** | 提供接口来创建一系列相关或相互依赖的对象 | 工厂的工厂 | 跨平台 UI 组件（Windows/Mac 风格） |
| **建造者 (Builder)** | 将复杂对象的构建过程与表示分离，同样的构建过程可以创建不同表示 | 分步骤构建 | 复杂配置对象（如 SQL 查询构建） |

### 6.2 结构型模式

结构型模式关注类和对象的组合方式，目标是形成更大的结构，同时保持结构的灵活和高效。

| 模式 | 目的 | 核心思想 | 典型场景 |
|------|------|----------|----------|
| **适配器 (Adapter)** | 将一个类的接口转换为客户端期望的另一个接口 | 包装不兼容的类 | 旧系统接口适配新系统、不同数据格式转换 |
| **装饰器 (Decorator)** | 动态地给对象添加额外功能，比继承更灵活 | 包装原对象，增加新行为 | Java I/O 流（BufferedInputStream 装饰 FileInputStream） |
| **代理 (Proxy)** | 为另一个对象提供代替品或占位符，以控制对该对象的访问 | 访问控制 | 延迟加载（虚拟代理）、权限控制（保护代理）、远程调用（远程代理） |
| **外观 (Facade)** | 为复杂子系统提供一个简化的统一接口 | 隐藏内部复杂性 | SDK/API 封装、简化第三方库调用 |

### 6.3 行为型模式

行为型模式关注对象之间的职责分配和算法选择，目标是使对象之间的交互更灵活。

| 模式 | 目的 | 核心思想 | 典型场景 |
|------|------|----------|----------|
| **观察者 (Observer)** | 定义一对多依赖关系，当一个对象状态改变时，所有依赖者自动收到通知 | 发布-订阅 | 事件系统、MVC 中 Model 变化通知 View、消息通知 |
| **策略 (Strategy)** | 定义一系列算法，把它们封装起来，使它们可以互换 | 将算法独立于使用它的客户端 | 排序策略选择、不同支付方式、不同压缩算法 |
| **模板方法 (Template Method)** | 定义算法骨架，将某些步骤延迟到子类实现 | 父类定义流程，子类实现细节 | 框架的生命周期方法（如 JUnit 的 setUp/tearDown） |
| **迭代器 (Iterator)** | 提供一种方法顺序访问聚合对象的元素，而不暴露底层表示 | 统一遍历接口 | C++ STL 迭代器、Java Iterator |
| **状态 (State)** | 允许对象在内部状态改变时改变其行为 | 用状态对象替代条件分支 | 有限状态机（如 TCP 连接状态、订单状态流转） |
| **命令 (Command)** | 将请求封装为对象，支持排队、日志、撤销操作 | 将"做什么"与"何时做"解耦 | 撤销/重做功能、事务、宏命令 |

### 6.4 设计模式总结

使用设计模式时应注意：
- **不要为了使用模式而使用模式**。设计模式是解决特定问题的方案，如果没有那个问题，就不需要那个模式
- **简单问题用简单方案**。如果一个 if-else 就能清晰解决的问题，不需要引入策略模式
- **模式之间可以组合**。例如 MVC 中同时使用了观察者模式（Model-View）和策略模式（Controller 选择处理策略）
