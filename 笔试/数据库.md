# 数据库

## 一、关系模型

### 1.1 基本概念

- **关系 (Relation)**：一张二维表
- **元组 (Tuple)**：表中一行
- **属性 (Attribute)**：表中一列
- **域 (Domain)**：属性的取值范围
- **码 (Key)**：
  - **候选码**：能唯一标识元组的最小属性集
  - **主码 (Primary Key)**：选定的一个候选码
  - **外码 (Foreign Key)**：引用其他关系主码的属性

### 1.2 关系代数

| 运算 | 符号 | 含义 |
|------|------|------|
| 选择 (Selection) | σ | 选取满足条件的行 |
| 投影 (Projection) | π | 选取指定列 |
| 并 (Union) | ∪ | 两关系的并集 |
| 差 (Difference) | − | 关系差 |
| 笛卡尔积 | × | 所有组合 |
| 连接 (Join) | ⋈ | 条件连接/自然连接 |
| 除 (Division) | ÷ | "全部包含"语义 |

**自然连接**：在公共属性上等值连接并去重。

**除运算**：R ÷ S 的含义——在 R 中找出与 S 的所有元组都有关联的那些元组在非公共属性上的投影。

## 二、SQL

### 2.1 DDL (数据定义)

```sql
CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    age INT CHECK (age > 0),
    dept_id INT REFERENCES departments(id)
);

ALTER TABLE students ADD COLUMN email VARCHAR(100);
DROP TABLE students;
```

### 2.2 DML (数据操作)

```sql
-- 查询
SELECT DISTINCT dept, COUNT(*) AS cnt
FROM students
WHERE age > 18
GROUP BY dept
HAVING COUNT(*) > 5
ORDER BY cnt DESC
LIMIT 10;

-- 连接查询
SELECT s.name, d.dept_name
FROM students s
INNER JOIN departments d ON s.dept_id = d.id;

-- 子查询
SELECT name FROM students
WHERE dept_id IN (SELECT id FROM departments WHERE location = 'Beijing');

-- 相关子查询
SELECT name FROM students s
WHERE age > (SELECT AVG(age) FROM students WHERE dept_id = s.dept_id);

-- 插入/更新/删除
INSERT INTO students (id, name, age) VALUES (1, 'Alice', 20);
UPDATE students SET age = 21 WHERE id = 1;
DELETE FROM students WHERE id = 1;
```

### 2.3 连接类型

| 连接 | 含义 |
|------|------|
| INNER JOIN | 只返回匹配的行 |
| LEFT JOIN | 返回左表全部 + 匹配的右表行 |
| RIGHT JOIN | 返回右表全部 + 匹配的左表行 |
| FULL OUTER JOIN | 返回两表全部行 |
| CROSS JOIN | 笛卡尔积 |
| SELF JOIN | 表与自身连接 |

### 2.4 视图 (View)

```sql
CREATE VIEW top_students AS
SELECT name, gpa FROM students WHERE gpa > 3.5;
```

- 虚拟表，基于查询定义，不存储数据
- 优点：简化查询、安全性（隐藏数据）、逻辑独立性
- 可更新视图有限制（不能有聚合、DISTINCT、GROUP BY 等）

### 2.5 索引

```sql
CREATE INDEX idx_name ON students(name);
CREATE UNIQUE INDEX idx_id ON students(id);
CREATE INDEX idx_composite ON students(dept_id, age);  -- 复合索引
```

## 三、数据库设计

### 3.1 E-R 模型

- **实体 (Entity)**：矩形
- **属性 (Attribute)**：椭圆
- **联系 (Relationship)**：菱形
- 联系类型：1:1、1:N、M:N

E-R 图转关系模型规则：
- 每个实体转为一个关系
- 1:1 联系：合并到任一实体或单独建表
- 1:N 联系：外码放在 N 端
- M:N 联系：必须单独建表

### 3.2 函数依赖

- **函数依赖 (FD)**：X → Y，X 的值能唯一确定 Y 的值
- **完全函数依赖**：X 的任何真子集都不能决定 Y
- **部分函数依赖**：X 的某个真子集就能决定 Y
- **传递函数依赖**：X → Y，Y → Z，且 Y ↛ X，则 X 传递决定 Z

**Armstrong 公理**：
1. **自反律**：若 Y ⊆ X，则 X → Y
2. **增广律**：若 X → Y，则 XZ → YZ
3. **传递律**：若 X → Y，Y → Z，则 X → Z

推导规则：合并律、分解律、伪传递律

**属性闭包**：给定 FD 集 F，X 关于 F 的闭包 X⁺ = 所有 X 能决定的属性。判断 X 是否为候选码：检查 X⁺ 是否包含全部属性。

### 3.3 范式

| 范式 | 要求 | 消除的问题 |
|------|------|-----------|
| 1NF | 每个属性不可再分（原子性） | — |
| 2NF | 1NF + 消除非主属性对码的部分函数依赖 | 部分依赖 |
| 3NF | 2NF + 消除非主属性对码的传递函数依赖 | 传递依赖 |
| BCNF | 所有函数依赖 X → Y 中，X 都是超码 | 主属性的依赖问题 |

**判断范式的方法**：
1. 找出所有候选码
2. 判断是否存在部分依赖（→ 非 2NF）
3. 判断是否存在传递依赖（→ 非 3NF）
4. 判断是否所有决定因素都是超码（→ 非 BCNF）

## 四、存储与索引

### 4.1 B 树 vs B+ 树

| 对比 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点存储数据 | 只有叶子节点存储数据 |
| 叶子链表 | 无 | 叶子节点形成有序链表 |
| 范围查询 | 需要中序遍历 | 沿叶子链表扫描，高效 |
| 查询稳定性 | 不稳定（可能在内部节点命中） | 稳定（总是到叶子节点） |

**B+ 树适合做数据库索引的原因**：
1. 非叶节点不存数据 → 每个节点能存更多 key → 树更矮 → 磁盘 I/O 少
2. 叶子链表 → 支持高效范围查询
3. 查询路径等长 → 性能稳定

### 4.2 索引类型

| 类型 | 特点 |
|------|------|
| 聚簇索引 (Clustered) | 数据行按索引顺序物理存储，每表只有一个 |
| 非聚簇索引 (Non-Clustered) | 索引和数据分开存储，叶子存指针 |
| Hash 索引 | 等值查询 O(1)，不支持范围查询 |
| 全文索引 | 文本搜索 |
| 复合索引 | 多列索引，遵循最左前缀原则 |

**最左前缀原则**：复合索引 (a, b, c)，查询条件中必须包含 a 才能使用索引。可匹配 (a)、(a,b)、(a,b,c)，不能匹配 (b,c)。

### 4.3 索引优化

- 在 WHERE、JOIN、ORDER BY、GROUP BY 涉及的列上建索引
- 避免在索引列上使用函数或运算
- 避免 SELECT *
- 覆盖索引（查询的列都在索引中，无需回表）
- EXPLAIN 分析执行计划

## 五、查询优化

### 5.1 查询执行计划

通过 `EXPLAIN` 查看。关注：
- **type**：访问类型，从好到差：const > eq_ref > ref > range > index > ALL
- **key**：使用的索引
- **rows**：预估扫描行数
- **Extra**：Using index(覆盖索引)、Using filesort(额外排序)、Using temporary(临时表)

### 5.2 等价变换规则

- 选择下推：尽早执行选择操作，减少中间结果
- 投影下推：尽早去除不需要的列
- 连接顺序优化：选择中间结果最小的连接顺序

## 六、事务管理

### 6.1 ACID

| 特性 | 含义 | 实现手段 |
|------|------|----------|
| **A (Atomicity)** 原子性 | 事务要么全部完成，要么全部回滚 | Undo Log |
| **C (Consistency)** 一致性 | 事务前后数据库一致性约束不变 | 应用层 + 数据库约束 |
| **I (Isolation)** 隔离性 | 并发事务互不干扰 | 锁 + MVCC |
| **D (Durability)** 持久性 | 事务提交后永久生效 | Redo Log |

### 6.2 事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现 |
|----------|------|-----------|------|------|
| 读未提交 (Read Uncommitted) | 可能 | 可能 | 可能 | 无锁 |
| 读已提交 (Read Committed) | 不可能 | 可能 | 可能 | 每次读取新快照 |
| 可重复读 (Repeatable Read) | 不可能 | 不可能 | 可能* | 事务开始时快照 |
| 串行化 (Serializable) | 不可能 | 不可能 | 不可能 | 完全串行/严格2PL |

*MySQL InnoDB 的 RR 级别通过 Next-Key Lock 在很大程度上避免了幻读。

**并发问题**：
- **脏读**：读到其他事务未提交的数据
- **不可重复读**：同一事务内两次读同一数据，结果不同（被其他事务修改）
- **幻读**：同一事务内两次查询，结果集行数不同（被其他事务插入/删除）

## 七、并发控制

### 7.1 锁机制

| 锁类型 | 描述 |
|--------|------|
| 共享锁 (S Lock / 读锁) | 多个事务可同时持有，读取数据 |
| 排他锁 (X Lock / 写锁) | 独占，读写数据 |
| 意向共享锁 (IS) | 表级锁，表示事务要对某行加 S 锁 |
| 意向排他锁 (IX) | 表级锁，表示事务要对某行加 X 锁 |

**锁兼容矩阵**：

|  | S | X |
|--|---|---|
| S | 兼容 | 冲突 |
| X | 冲突 | 冲突 |

### 7.2 两阶段锁协议 (2PL)

- **扩展阶段**：只能加锁，不能解锁
- **收缩阶段**：只能解锁，不能加锁
- 保证可串行化，但不能避免死锁
- **严格 2PL**：所有写锁在事务结束时才释放 → 避免级联回滚

### 7.3 死锁处理

- **预防**：一次封锁法（一次申请全部锁）、顺序封锁法
- **检测**：等待图（检测是否有环）
- **超时**：等待超时则回滚
- **恢复**：选择代价最小的事务回滚

### 7.4 MVCC (多版本并发控制)

核心思想：为每个数据保存多个版本，读操作读取快照版本，不加锁，实现读写不冲突。

InnoDB 实现：
- 每行记录有隐藏列：DB_TRX_ID（最近修改的事务ID）、DB_ROLL_PTR（指向 Undo Log 的指针）
- **Read View**：事务读取时生成，包含当前活跃事务ID列表
- 根据版本链和 Read View 判断数据可见性

## 八、恢复技术

### 8.1 日志

- **Undo Log**：记录修改前的值，用于回滚（保证原子性）
- **Redo Log**：记录修改后的值，用于重做（保证持久性）
- **WAL (Write-Ahead Logging)**：先写日志，再写数据

### 8.2 检查点 (Checkpoint)

定期将内存中的修改刷入磁盘并记录检查点。恢复时只需从最近的检查点开始，减少恢复时间。

### 8.3 恢复过程

1. 从检查点开始扫描日志
2. 建立 Redo 列表（已提交事务）和 Undo 列表（未提交事务）
3. 对 Redo 列表中的事务重做
4. 对 Undo 列表中的事务回滚

## 九、NoSQL

### 9.1 CAP 定理

分布式系统最多同时满足以下三项中的两项：
- **C (Consistency)**：一致性，所有节点在同一时刻看到相同数据
- **A (Availability)**：可用性，每个请求都能收到非错误响应
- **P (Partition Tolerance)**：分区容忍性，网络分区时系统仍能运作

实际中 P 必须保证，所以是在 **CP** 和 **AP** 之间选择。

### 9.2 BASE 理论

- **B**asically Available（基本可用）
- **S**oft State（软状态）
- **E**ventually Consistent（最终一致性）

是对 CAP 中 AP 的延伸，追求最终一致性而非强一致性。

### 9.3 NoSQL 类型

| 类型 | 特点 | 代表 |
|------|------|------|
| KV 存储 | 键值对，简单高效 | Redis, Memcached |
| 文档数据库 | JSON/BSON 文档 | MongoDB, CouchDB |
| 列族数据库 | 按列存储，适合分析 | HBase, Cassandra |
| 图数据库 | 节点+边，关系查询 | Neo4j, JanusGraph |
