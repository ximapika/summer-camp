# 编译原理

编译器是将高级语言源程序翻译为目标机器语言程序的软件。理解编译原理有助于深入理解程序如何从人类可读的代码变成 CPU 可执行的指令，也有助于理解编程语言的设计决策和各种语言特性的实现原理。

## 一、编译过程概述

一个完整的编译过程分为六个阶段，可以从逻辑上分为前端和后端：

```
源程序 → 词法分析 → 语法分析 → 语义分析 → 中间代码生成 → 代码优化 → 目标代码生成 → 目标程序
         (Token流)   (语法树)    (标注语法树)  (三地址码等)    (优化后IR)    (汇编/机器码)
```

**前端**（与目标机器无关）：
- **词法分析**：将字符流分割成有意义的单词（Token），如标识符、关键字、运算符、常量
- **语法分析**：根据文法规则将 Token 序列组织成语法树，检查程序的语法结构是否正确
- **语义分析**：检查语法树的语义是否合法（如类型检查、变量是否声明），并附加语义信息
- **中间代码生成**：将语法树转换为与机器无关的中间表示（如三地址码）

**后端**（与目标机器相关）：
- **代码优化**：对中间代码进行各种变换以提高执行效率
- **目标代码生成**：将优化后的中间代码翻译为目标机器的汇编或机器指令

**符号表 (Symbol Table)**：贯穿编译全过程的核心数据结构。记录源程序中每个标识符（变量、函数、类型等）的名称、类型、作用域、存储地址等信息。各个阶段都会查询和更新符号表。

**错误处理**：同样贯穿始终。词法错误（非法字符）、语法错误（括号不匹配）、语义错误（类型不匹配）应尽可能被检测到并给出有意义的错误信息。

## 二、词法分析

词法分析器（也称扫描器/lexer/tokenizer）负责读入源程序的字符流，识别出一个个有意义的**词法单元 (Token)**。例如 `int x = 42 + y;` 会被分割为：`<关键字, int>` `<标识符, x>` `<运算符, =>` `<整数常量, 42>` `<运算符, +>` `<标识符, y>` `<分号, ;>`。

词法分析的理论基础是正则表达式和有限自动机。

### 2.1 正则表达式

正则表达式是描述字符串模式的工具。每种 Token（标识符、数字、关键字等）都可以用正则表达式定义。例如，C 语言的标识符可以用 `[a-zA-Z_][a-zA-Z_0-9]*` 描述。

| 运算 | 含义 | 优先级 | 示例 |
|------|------|--------|------|
| `a\|b` | 选择（或）：匹配 a 或 b | 低 | `if\|else` 匹配 "if" 或 "else" |
| `ab` | 连接：先匹配 a 再匹配 b | 中 | `ab` 匹配 "ab" |
| `a*` | Kleene 闭包：a 重复 0 次或多次 | 高 | `a*` 匹配 "", "a", "aa", ... |
| `a+` | 正闭包：a 重复 1 次或多次，等价于 `aa*` | 高 | `[0-9]+` 匹配至少一位数字 |
| `a?` | 可选：a 出现 0 次或 1 次 | 高 | `-?[0-9]+` 匹配可选负号的整数 |

正则表达式的表达能力是有限的——它不能描述嵌套结构（如匹配括号），这就需要更强大的上下文无关文法来描述程序的语法结构。

### 2.2 有限自动机

有限自动机 (Finite Automaton) 是识别正则语言的计算模型。词法分析器的本质就是一个有限自动机——根据当前状态和输入字符，转移到下一个状态，到达接受状态时就识别出一个 Token。

**NFA (非确定有限自动机, Nondeterministic Finite Automaton)**：
- 允许 **ε 转换**（不消耗输入字符的转换）
- 一个状态对同一输入可以转向**多个**不同状态（非确定性）
- 每个正则表达式都可以通过 **Thompson 构造法**直接转换为 NFA
- 结构简单，但直接模拟效率不高

**DFA (确定有限自动机, Deterministic Finite Automaton)**：
- **无** ε 转换
- 一个状态对每个输入符号**恰好**有一个转换（确定性）
- 识别效率高（每个输入字符只需一步转换），是实际词法分析器使用的模型
- NFA 和 DFA 的表达能力相同——它们能识别的语言集合是一样的（都是正则语言）

### 2.3 NFA → DFA（子集构造法）

子集构造法将 NFA 转换为等价的 DFA。核心思想是：DFA 的每个状态对应 NFA 的一个**状态集合**。

1. 计算 NFA 起始状态的 **ε-closure**（从起始状态出发，只通过 ε 转换能到达的所有状态的集合）→ 这就是 DFA 的起始状态
2. 对 DFA 的每个状态（NFA 状态集合），对每个输入符号 a：
   - 计算 move(状态集, a)：所有能通过 a 转换到达的 NFA 状态
   - 再计算这些状态的 ε-closure → 得到新的 DFA 状态
3. 重复步骤 2，直到不再产生新的 DFA 状态
4. DFA 中包含 NFA 终态的状态集合被标记为 DFA 的终态

**注意**：子集构造可能导致状态数指数级增长（NFA 有 n 个状态，DFA 最多 2^n 个状态），但实际中通常远小于此。

### 2.4 DFA 最小化

不同的 DFA 可能识别同一种语言。DFA 最小化的目标是找到状态数最少的等价 DFA。算法基于**状态等价**的概念——如果两个状态对所有输入串都产生相同的接受/拒绝结果，则它们等价，可以合并。

**Hopcroft 算法**：
1. 初始划分：将所有状态分为**终态组**和**非终态组**
2. 对每个组内的状态检查：是否存在某个输入符号使组内两个状态转向**不同组**？如果是，将该组分裂
3. 重复步骤 2，直到不可再分
4. 每个组合并为 DFA 的一个状态

## 三、语法分析

词法分析处理的是线性的 Token 序列，语法分析则需要识别 Token 之间的**层次结构**——例如，表达式 `a + b * c` 中乘法应该先于加法计算。语法分析器（parser）根据文法规则将 Token 序列组织成**语法树 (Parse Tree / Syntax Tree)**，同时检查程序的语法是否合法。

### 3.1 上下文无关文法 (CFG)

上下文无关文法 (Context-Free Grammar) 是描述程序语言语法结构的标准工具。一个 CFG 由四元组 G = (V, T, P, S) 定义：
- **V**：非终结符集合（代表语法结构，如"表达式"、"语句"）
- **T**：终结符集合（实际的 Token，如 `+`、`if`、标识符）
- **P**：产生式集合（语法规则，如 E → E + T | T）
- **S**：起始符号（表示整个程序的顶层结构）

**关键概念**：
- **推导 (Derivation)**：从起始符号出发，反复用产生式右部替换左部的非终结符，直到得到全是终结符的串。**最左推导**每次替换最左边的非终结符；**最右推导（规范推导）**每次替换最右边的
- **语法树 (Parse Tree)**：展示推导过程的树结构。根节点是起始符号，叶节点是终结符，内部节点是非终结符
- **二义性 (Ambiguity)**：如果某个句子存在两棵不同的语法树（即两种不同的最左推导），则该文法是**二义的**。二义文法不能直接用于自动语法分析，需要通过改写文法或引入优先级/结合性规则来消除

### 3.2 文法变换

某些文法形式不适合特定的分析方法，需要进行变换。

**消除左递归**：
LL 分析（自顶向下）不能处理左递归文法（如 E → E + T），因为会导致无限递归。需要将左递归改写为右递归：

- 直接左递归：A → Aα | β 变为：
  - A → βA'
  - A' → αA' | ε
- 间接左递归（如 A → Bα, B → Aβ）：先通过代入消除间接关系，再消除直接左递归

**提取左公因子**：
当两个产生式有相同的前缀时（如 A → αβ₁ | αβ₂），分析器在只看到 α 时无法决定选哪个。提取公因子：
- A → αA'
- A' → β₁ | β₂

### 3.3 自顶向下分析

从起始符号出发，尝试推导出输入串。核心问题是：当前非终结符有多个产生式时，选择哪一个？

**递归下降分析**：为每个非终结符写一个递归函数。根据当前输入 Token（向前看符号）选择产生式。简单直观，许多编译器手写 parser 采用此方式（如 GCC）。

**LL(1) 分析**：
- 第一个 **L**：从**左**向右扫描输入
- 第二个 **L**：产生**最左**推导
- **1**：向前看 **1** 个符号来决定选择哪个产生式

LL(1) 是最简单的表驱动自顶向下分析方法。构造 LL(1) 分析器需要计算两个重要的集合：

**FIRST 集**：FIRST(α) = 从 α 能推导出的所有串的**首终结符**的集合
- 如果 α 是终结符 a，则 FIRST(α) = {a}
- 如果 α →* ε（α 可以推导出空串），则 ε ∈ FIRST(α)
- 如果 α = X₁X₂...Xₙ，则 FIRST(α) 包含 FIRST(X₁) 中非 ε 的部分；如果 ε ∈ FIRST(X₁)，还要加入 FIRST(X₂) 中非 ε 的部分，以此类推

**FOLLOW 集**：FOLLOW(A) = 在所有句型中紧跟在非终结符 A **之后**的终结符的集合
- 如果 A 是起始符号，则 $ ∈ FOLLOW(A)（$ 表示输入结束标记）
- 如果 B → αAβ，则 FIRST(β) - {ε} ⊆ FOLLOW(A)
- 如果 B → αA，或 B → αAβ 且 ε ∈ FIRST(β)，则 FOLLOW(B) ⊆ FOLLOW(A)

**LL(1) 分析表构造**：
- 对每个产生式 A → α：
  - 对 FIRST(α) 中每个终结符 a，将 A → α 放入 M[A, a]
  - 若 ε ∈ FIRST(α)，对 FOLLOW(A) 中每个终结符 b，将 A → α 放入 M[A, b]
- **无冲突**（每个格子最多一个产生式）则文法是 LL(1) 的

### 3.4 自底向上分析

从输入串出发，逐步**归约**到起始符号。与自顶向下相反，自底向上分析实质上是最右推导的**逆过程**（最左归约）。

**LR 分析**是最强大的自底向上分析方法：
- **L**：从**左**向右扫描输入
- **R**：构造**最右**推导（的逆序）

**核心操作**：
- **移入 (Shift)**：将下一个输入 Token 压入分析栈
- **归约 (Reduce)**：将栈顶与某个产生式右部匹配的符号序列（**句柄**）替换为产生式左部的非终结符
- 分析器在每一步根据当前栈顶状态和向前看符号，查 **ACTION 表**决定移入还是归约；查 **GOTO 表**确定归约后的状态转移

**LR(0) 项目**：在产生式中加一个**点 (·)** 表示分析进度。例如 A → α·β 表示 α 已在栈中、β 还待输入。点在最右边（A → αβ·）表示可以归约。

| 分析方法 | 分析能力 | 特点 |
|----------|----------|------|
| **LR(0)** | 最弱 | 不看向前符号就决定动作，容易有冲突 |
| **SLR(1)** | 较弱 | 用 FOLLOW 集解决冲突，但 FOLLOW 集可能过大 |
| **LR(1)** | 最强 | 每个项目附带精确的展望符（lookahead），能区分更多情况 |
| **LALR(1)** | 介于 SLR 和 LR(1) 之间 | 合并 LR(1) 中"核心相同"的项目集，大幅减少状态数。**yacc/bison** 使用此方法 |

**冲突类型**：
- **移入-归约冲突**：不确定应该移入下一个符号还是立即归约
- **归约-归约冲突**：不确定应该按哪个产生式进行归约

## 四、语义分析

语法分析只检查程序的结构是否符合文法规则，但不检查语义是否合理。例如 `int x = "hello"` 语法上可能合法（赋值语句的结构正确），但语义上类型不匹配。语义分析的主要任务包括**类型检查**、**变量声明检查**和**作用域分析**。

### 4.1 属性文法

属性文法在 CFG 的基础上，为文法符号附加**属性**（如类型、值、地址），并定义**语义规则**来计算这些属性。这是语义分析的形式化框架。

- **综合属性 (Synthesized Attribute)**：从子节点的属性计算得到，信息**自底向上**传递。例如表达式的值可以从子表达式的值计算
- **继承属性 (Inherited Attribute)**：从父节点或兄弟节点的属性计算得到，信息**自顶向下**传递。例如声明语句中的类型信息从左向右传递给各个变量

**属性文法的分类**：
- **S-属性文法**：只包含综合属性。可以在 LR 分析过程中自底向上地计算
- **L-属性文法**：继承属性只依赖于父节点的继承属性和左兄弟节点的属性。可以在一次从左到右的遍历中计算

### 4.2 语法制导翻译 (SDT)

语法制导翻译在产生式中嵌入**语义动作**（代码片段），在语法分析的过程中执行这些动作来完成翻译。这是编译器前端实际使用的技术——yacc/bison 中产生式后面的 `{ ... }` 就是语义动作。

### 4.3 类型检查

类型检查是语义分析的核心任务，确保运算和操作的类型是合法的。

- **静态类型检查**：在编译时进行类型检查。编译器在分析阶段就能发现类型错误（C, C++, Java, Go, Rust）
- **动态类型检查**：在运行时进行类型检查。变量没有固定类型，值才有类型（Python, JavaScript, Ruby）
- **类型推断 (Type Inference)**：编译器自动推导变量的类型，程序员不必显式声明（ML, Haskell, C++ 的 `auto`, Rust 的类型推断）
- **类型转换**：隐式转换（如 int → float，称为类型提升/coercion）和显式转换（强制类型转换/cast）

## 五、中间代码

中间代码（Intermediate Representation, IR）是介于源语言和目标语言之间的表示形式。使用中间代码的好处是**前后端分离**：前端将不同源语言翻译为同一种 IR，后端将 IR 翻译为不同目标机器的代码。这样 m 种源语言 × n 种目标机器只需 m + n 个翻译器，而非 m × n 个。

### 5.1 三地址码

三地址码是最常用的中间表示形式，每条指令最多包含一个运算符和三个操作数（两个源操作数和一个目的操作数），形如 `x = y op z`。

```
// 源代码：d = a + b * c
t1 = b * c       // 临时变量 t1 存储 b*c 的结果
t2 = a + t1      // 临时变量 t2 存储 a+t1 的结果
d = t2            // 赋值
```

### 5.2 四元式

四元式是三地址码的一种具体表示方式：(运算符, 操作数1, 操作数2, 结果)

```
(*, b, c, t1)     // t1 = b * c
(+, a, t1, t2)    // t2 = a + t1
(=, t2, _, d)     // d = t2
```

### 5.3 语法树与 DAG

- **抽象语法树 (AST)**：源程序的树形中间表示，比具体语法树（parse tree）更紧凑——省略了不携带语义信息的节点（如括号、关键字）
- **有向无环图 (DAG)**：在 AST 基础上**合并公共子表达式**。如果 `a + b` 在表达式中出现两次，DAG 中只计算一次并共享结果节点。DAG 可以直接用于公共子表达式消除优化

### 5.4 静态单赋值形式 (SSA)

SSA (Static Single Assignment) 是现代编译器广泛使用的 IR 形式。核心规则：每个变量只被赋值一次。当同一个变量有多个赋值时，重命名为不同的版本：

```
// 原始代码           // SSA 形式
x = 1               x1 = 1
x = x + 1           x2 = x1 + 1
y = x + 2           y1 = x2 + 2
```

SSA 使得数据流分析和许多优化变得更简单高效（如常量传播、死代码消除）。LLVM IR 就是基于 SSA 的。

## 六、代码优化

代码优化的目标是在不改变程序语义的前提下，提高目标代码的执行效率（速度和/或空间）。优化可以在不同级别进行。

### 6.1 局部优化（基本块内）

**基本块 (Basic Block)** 是一段顺序执行的代码，只有一个入口（第一条语句）和一个出口（最后一条语句），中间没有分支。局部优化在单个基本块内进行，相对简单但效果显著。

| 优化技术 | 描述 | 示例 |
|----------|------|------|
| **常量折叠** | 编译时直接计算常量表达式的值 | `x = 2 + 3` → `x = 5` |
| **常量传播** | 如果变量的值是已知常量，在使用处替换为常量 | `x = 5; y = x + 1` → `y = 6` |
| **公共子表达式消除** | 如果同一表达式已经计算过，直接使用之前的结果 | `t1 = a+b; t2 = a+b` → `t2 = t1` |
| **死代码删除** | 删除计算结果永远不会被使用的语句 | 赋值给一个之后不再使用的变量 |
| **复写传播** | 赋值 `x = y` 后，将后续对 x 的使用替换为 y | `x = y; z = x+1` → `z = y+1` |
| **代数化简** | 利用代数恒等式简化运算 | `x*1 → x`、`x+0 → x`、`x*2 → x<<1` |

### 6.2 循环优化

程序的大部分执行时间花在循环中（90/10 法则：90% 的时间花在 10% 的代码上），因此循环优化对性能影响最大。

| 优化技术 | 描述 | 示例 |
|----------|------|------|
| **代码外提 (Loop-Invariant Code Motion)** | 将循环中不随迭代变化的计算移到循环外 | `for(i) { x = a*b; ... }` → `x = a*b; for(i) { ... }` |
| **强度削减 (Strength Reduction)** | 将循环中的高代价运算替换为低代价运算 | 循环中的 `i*4` → 每次迭代 `+4`（乘法→加法） |
| **归纳变量消除** | 消除只用于循环控制且可由其他归纳变量推导的变量 | 两个同步递增的变量保留一个 |
| **循环展开 (Loop Unrolling)** | 将循环体复制多份以减少循环控制开销和增加指令级并行 | 循环体执行 4 次合并为 1 次迭代 |
| **循环融合** | 将两个遍历相同范围的循环合并 | 减少循环开销和提高缓存利用率 |

### 6.3 数据流分析

数据流分析是全局优化的基础。它通过分析程序中数据如何在各语句间流动，为优化提供必要的信息。

- **到达定值分析 (Reaching Definitions)**：对于程序中某一点，哪些赋值语句的值可以到达这里（没有被中间的重新赋值覆盖）。用于常量传播和公共子表达式消除
- **活跃变量分析 (Live Variable Analysis)**：在程序某一点之后，某个变量是否还会被使用？如果不会（该变量"死亡"），则对它的赋值就是死代码，可以删除。也用于寄存器分配
- **可用表达式分析 (Available Expressions)**：在程序某一点，哪些表达式的值已经计算过且没有被修改？这些表达式可以直接复用，不需要重新计算。用于公共子表达式消除

数据流分析通常在程序的**控制流图 (CFG, Control Flow Graph)** 上进行。CFG 的节点是基本块，边表示可能的控制流转移。分析通过在 CFG 上反复迭代直到达到不动点来完成。

## 七、运行时环境

编译器不仅要翻译代码，还要为目标程序的运行时行为做安排——如何分配内存、如何传递参数、如何管理函数调用。

### 7.1 栈式存储分配

每次函数调用都会在运行时栈上创建一个**活动记录 (Activation Record)** / **栈帧 (Stack Frame)**，函数返回时销毁。这就是为什么局部变量在函数返回后就不存在了。

```
┌─────────────────┐ ← 高地址
│   实参           │   调用者传入的参数
│   返回地址        │   函数返回后要回到的地址
│   旧的帧指针(FP) │ ← 当前 FP 指向这里，用于恢复调用者的栈帧
│   局部变量        │   函数的局部变量
│   临时变量        │   编译器生成的临时值
│   保存的寄存器    │   被调用者保存的寄存器值（callee-saved）
└─────────────────┘ ← SP（栈顶）低地址
```

**调用过程**：
1. 调用者将参数压栈（或放入寄存器），执行 CALL 指令（压入返回地址并跳转）
2. 被调用者保存旧 FP，设置新 FP，为局部变量分配空间（SP 下移）
3. 执行函数体
4. 函数返回：恢复 SP 和 FP，RET 指令（弹出返回地址并跳转回调用者）

### 7.2 堆管理

堆（Heap）用于存放动态分配的数据（C 的 malloc/free，C++ 的 new/delete，Java 的 new）。与栈不同，堆上的内存生命周期不与函数调用绑定——程序员（或垃圾回收器）显式管理。

**垃圾回收 (Garbage Collection, GC)**：自动回收不再使用的堆内存（Java, Go, Python 等语言）。
- **引用计数 (Reference Counting)**：每个对象维护一个引用计数，为 0 时回收。简单但无法处理循环引用
- **标记-清除 (Mark-Sweep)**：从根（栈、全局变量）出发标记所有可达对象，清除未标记的。会产生内存碎片
- **标记-整理 (Mark-Compact)**：标记后将存活对象移到一端，消除碎片。但移动对象代价大
- **分代收集 (Generational GC)**：基于"大部分对象很快死亡"的观察，将堆分为新生代和老年代，新生代频繁回收，老年代偶尔回收。JVM 采用此策略

### 7.3 作用域与绑定

**作用域 (Scope)** 决定了一个名字（变量、函数）在程序中哪些位置可以被引用。

- **静态作用域（词法作用域, Lexical Scope）**：变量的作用域在**编译时**由其在源代码中的位置决定。内层可以访问外层的变量。绝大多数语言采用此方式（C, Java, Python, JavaScript）
- **动态作用域 (Dynamic Scope)**：变量的作用域在**运行时**由调用链决定。函数中引用的自由变量在调用者的作用域中查找，而非定义处的作用域。只有少数语言采用（bash、早期 Lisp、Perl 的 `local`）

**绑定 (Binding)**：将名字与实体（变量、函数、类型）关联起来。绑定时机不同：
- 编译时绑定（静态绑定）：类型检查、地址分配
- 运行时绑定（动态绑定）：虚函数调用、动态类型语言的方法分派
