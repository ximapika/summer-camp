# 计算机组成原理

计算机组成原理研究计算机硬件系统的基本组成和工作原理。理解计组有助于编写更高效的代码——了解 Cache 就知道为什么按行遍历数组比按列快，了解流水线就知道为什么分支预测对性能影响巨大。

## 一、数据表示

计算机中所有的数据（数字、文字、图像、声音）最终都是用二进制（0 和 1）表示的。数值在计算机中有多种编码方式，不同编码方式决定了计算机如何进行算术运算。

### 1.1 数值编码

计算机中的带符号整数使用最高位作为符号位（0 正 1 负），但如何表示数值部分有多种编码方式。

**原码**：最直观的编码方式。符号位 + 绝对值的二进制表示。例如 +5 = 0101，-5 = 1101（4 位）。
- 缺点：加减法运算需要先判断符号再决定操作，+0 (0000) 和 -0 (1000) 有两种表示

**反码**：正数同原码，负数将原码除符号位外各位取反。例如 -5 = 1010。
- 仍然有 +0 和 -0 两种表示，主要作为求补码的中间步骤

**补码**：现代计算机实际使用的编码。正数同原码，负数 = 反码 + 1。例如 -5 = 1011。
- **核心优势**：0 的表示唯一（0000），加减法统一为加法运算（符号位参与运算），硬件实现简单
- 多表示一个负数：n 位补码范围为 [-2^(n-1), 2^(n-1)-1]，例如 8 位 [-128, 127]
- [-128] 的补码 = 1000 0000（没有对应的原码和反码）
- **补码的补码 = 原码**

**移码**：补码的符号位取反。例如 +5 的补码 0101 → 移码 1101；-5 的补码 1011 → 移码 0011。
- 特点：移码的数值大小顺序与无符号整数一致，**便于直接比较大小**
- 应用：IEEE 754 浮点数的阶码使用移码表示

| 编码 | 正数 | 负数 | 范围 (n位) | 零的表示 | 应用 |
|------|------|------|-----------|----------|------|
| 原码 | 0+数值 | 1+数值 | -(2^(n-1)-1) ~ 2^(n-1)-1 | +0 和 -0 | 乘除法 |
| 反码 | 同原码 | 符号位1，其余取反 | 同原码 | +0 和 -0 | 求补码的中间步骤 |
| 补码 | 同原码 | 反码+1 | -2^(n-1) ~ 2^(n-1)-1 | 唯一的 0 | 加减法、存储 |
| 移码 | 补码符号位取反 | 补码符号位取反 | 同补码 | 唯一 | 浮点数阶码 |

### 1.2 IEEE 754 浮点数

浮点数用于表示实数（带小数点的数），采用科学计数法的思想：一个数 = 符号 × 尾数 × 2^指数。

**格式**：`(-1)^S × 1.M × 2^(E-Bias)`

- S：符号位（0 正 1 负）
- E：阶码（指数部分），用移码表示（加上偏置值 Bias）
- M：尾数（有效数字部分），隐含前导 1（规格化数）

| 格式 | 总位数 | S | E(阶码) | M(尾数) | Bias | 有效位数 | 十进制精度 |
|------|--------|---|---------|---------|------|----------|-----------|
| 单精度 float | 32 | 1 | 8 | 23 | 127 | 24 位 | 约 7 位 |
| 双精度 double | 64 | 1 | 11 | 52 | 1023 | 53 位 | 约 15-16 位 |

**举例**：十进制 -6.5 的 float 表示
1. -6.5 = -110.1₂ = -1.101 × 2²
2. S = 1（负数），E = 2 + 127 = 129 = 10000001₂，M = 10100...0
3. 存储：1 10000001 10100000000000000000000

**特殊值**：阶码 E 全 0 或全 1 时表示特殊数值：

| 阶码 E | 尾数 M | 表示 | 含义 |
|--------|--------|------|------|
| 全 0 | 全 0 | ±0 | 正零和负零 |
| 全 0 | ≠ 0 | 非规格化数 | 用于表示极其接近 0 的小数（隐含位为 0 而非 1） |
| 全 1 | 全 0 | ±∞ | 正无穷和负无穷（如 1.0/0.0） |
| 全 1 | ≠ 0 | NaN | Not a Number（如 0.0/0.0、sqrt(-1)） |

**浮点运算步骤**：
1. **对阶**：将两个操作数的阶码调整为相同（小阶对大阶，避免高位丢失）
2. **尾数运算**：对齐后的尾数进行加减
3. **规格化**：调整结果使其满足 1.xxx 的形式（左规或右规）
4. **舍入**：因为尾数位数有限，需要舍入。常用方法：就近舍入（银行家舍入）、向零舍入
5. **溢出判断**：阶码超出范围则上溢（→ ±∞）或下溢（→ 0 或非规格化数）

**浮点数精度问题**：0.1 + 0.2 ≠ 0.3（在大多数编程语言中），因为 0.1 在二进制中是无限循环小数，存储时产生舍入误差。

### 1.3 溢出检测

整数运算结果超出表示范围时发生溢出。CPU 需要检测溢出以便处理（如触发异常）。

- **单符号位法**：两个同号数相加结果异号 → 溢出。具体地：正+正=负（正溢出）或 负+负=正（负溢出）
- **双符号位法（变形补码）**：用两位表示符号。正常结果符号位为 00(正) 或 11(负)；如果出现 01(正溢出) 或 10(负溢出) 则溢出。第一位是真正的符号位
- **进位判断法**：最高数据位向符号位的进位 C₁ 与符号位的进位 C₀ 不同时溢出。即 C₁ ⊕ C₀ = 1 → 溢出

## 二、运算器

运算器（也称 ALU，Arithmetic Logic Unit）是 CPU 中执行算术和逻辑运算的核心部件。

### 2.1 加法器

加法是最基本的运算，减法可以用加法实现（减去一个数 = 加上它的补码），乘法和除法也可以分解为移位和加法。因此加法器是运算器的核心。

- **串行进位加法器 (Ripple Carry Adder, RCA)**：n 位加法器由 n 个全加器级联，每个全加器的进位输出连接到下一个全加器的进位输入。进位逐级传递（"涟漪式"），总延迟与位数成正比 O(n)。结构简单但速度慢
- **超前进位加法器 (Carry Lookahead Adder, CLA)**：通过预先计算所有位的进位来加速。定义：
  - 生成函数：G_i = A_i · B_i（该位自己产生进位）
  - 传播函数：P_i = A_i ⊕ B_i（该位传播来自低位的进位）
  - 进位公式：C_i = G_i + P_i · C_{i-1}（展开后所有进位只依赖初始进位 C₀）
  - 所有位的进位可以并行计算，延迟 O(log n)

### 2.2 乘法器与除法器

- **原码一位乘法**：符号位单独异或确定，绝对值部分用"加法+右移"方式逐位相乘
- **补码一位乘法 (Booth 算法)**：直接对补码进行乘法，不需要先转换为原码。根据乘数**末两位**决定操作：
  - 00 或 11：只右移
  - 01：加被乘数，再右移
  - 10：减被乘数，再右移
  - Booth 算法可以处理负数乘法，且当乘数中有连续的 1 时效率更高
- **除法器**：
  - 原码恢复余数除法：试商后如果余数为负，恢复（加回除数）
  - 不恢复余数除法（加减交替法）：余数为正则左移后减除数，余数为负则左移后加除数。效率更高

### 2.3 ALU

算术逻辑单元是运算器的核心，由加法器 + 逻辑电路 + 多路选择器组成。能执行：
- **算术运算**：加、减、乘、除、增量、减量
- **逻辑运算**：与 (AND)、或 (OR)、非 (NOT)、异或 (XOR)
- **移位运算**：逻辑移位、算术移位、循环移位

ALU 通过控制信号选择执行哪种运算，并输出运算结果和状态标志（零标志 ZF、进位标志 CF、溢出标志 OF、符号标志 SF）。

## 三、存储系统

### 3.1 存储器层次结构

计算机使用多级存储层次来平衡速度、容量和成本之间的矛盾。越靠近 CPU 的存储越快、越贵、容量越小。

```
寄存器 ← 最快（< 1ns）、最贵、最小（几十~几百个）
  ↕ 
Cache（高速缓存）← 快（1-10ns）、SRAM、KB~MB 级
  ↕
主存（内存）← 较快（50-100ns）、DRAM、GB 级
  ↕
外存（磁盘/SSD）← 慢（ms 级/μs 级）、最便宜、TB 级
```

这种层次结构之所以有效，是因为**局部性原理**：
- **时间局部性**：刚被访问的数据很可能很快再次被访问（如循环变量）
- **空间局部性**：刚被访问的数据附近的数据很可能很快被访问（如数组遍历）

### 3.2 SRAM vs DRAM

主存和 Cache 使用不同的半导体存储技术。

| 对比 | SRAM (静态随机存储器) | DRAM (动态随机存储器) |
|------|------|------|
| 存储元件 | 双稳态触发器（通常 6 个晶体管） | 微小电容 + 1 个晶体管 (1T1C) |
| 速度 | 快（与 CPU 速度接近） | 较慢（需要充放电） |
| 刷新 | 不需要（触发器能保持状态） | **需要定期刷新**（电容会漏电，通常每 64ms 全部刷新一次） |
| 集成度 | 低（每个存储单元需要更多晶体管） | 高（结构简单，可以做得很密集） |
| 成本 | 高 | 低 |
| 功耗 | 低（静态功耗小） | 较高（刷新消耗能量） |
| 用途 | **Cache**（容量小但需要快） | **主存**（容量大但可以稍慢） |

### 3.3 Cache（高速缓存）

Cache 是 CPU 和主存之间的高速缓冲存储器，用于缓解 CPU 和主存之间巨大的速度差距。CPU 访问数据时先查 Cache，命中则直接获取（快），未命中则从主存加载到 Cache 再获取（慢）。由于局部性原理，Cache 的命中率通常 > 95%。

**映射方式**（决定主存中的块可以放到 Cache 的哪个位置）：

| 方式 | 原理 | 冲突概率 | 硬件复杂度 | 实际应用 |
|------|------|----------|-----------|----------|
| **直接映射** | 主存块号 % Cache 行数 = Cache 行号。每个主存块只能放在固定的一行 | 高（不同块竞争同一行） | 低（只需比较一个标记） | L1 Cache（对速度要求极高） |
| **全相联映射** | 主存块可以放在 Cache 的**任意一行** | 低（只有 Cache 满了才冲突） | 高（需要与所有行并行比较标记） | TLB（容量很小） |
| **组相联映射** | Cache 分为多个组，主存块号 % 组数 = 组号，块可以放在该组内的**任意一行** | 折中 | 折中 | **最常用**（L2/L3 Cache） |

- **n 路组相联**：每组有 n 行。n=1 退化为直接映射，n=Cache行数 退化为全相联。现代 CPU 通常用 4-16 路组相联

**Cache 地址结构**（以组相联为例）：
- 主存地址 = | 标记 (Tag) | 组号 (Index) | 块内偏移 (Offset) |
- 组号用于定位 Cache 组，标记用于在组内比较是否命中，偏移用于定位块内的具体字节

**替换算法**（Cache 满了需要换出哪一行）：

| 算法 | 策略 | 特点 |
|------|------|------|
| **Random** | 随机选择 | 简单，效果还行 |
| **FIFO** | 替换最先进入的行 | 简单，但可能替换常用行 |
| **LRU** | 替换**最近最久未使用**的行 | 效果最好，实现复杂（需要维护使用时间信息） |
| **LFU** | 替换**使用频率最低**的行 | 需要维护计数器 |

**写策略**（CPU 写数据时如何处理 Cache 和主存的一致性）：

| 策略 | 命中时 | 未命中时 | 特点 |
|------|--------|----------|------|
| **写直达 (Write Through)** | 同时写 Cache 和主存 | 写分配或非写分配 | 实现简单，主存总是最新的，但每次写都访问主存（慢） |
| **写回 (Write Back)** | 只写 Cache，设**脏位 (Dirty Bit)** | 通常写分配 | 减少主存写次数，但需要替换时回写脏块 |

- **写分配 (Write Allocate)**：写未命中时先把块从主存调入 Cache，再写 Cache。利用空间局部性
- **非写分配 (No Write Allocate)**：写未命中时直接写主存，不调入 Cache

现代 CPU 通常使用**写回 + 写分配**的组合。

### 3.4 主存与 CPU 的连接

当单个存储芯片的容量或字长不满足需求时，需要将多个芯片组合使用。

- **位扩展**：增加每个存储单元的位数。将多个芯片的数据引脚并联，地址线和控制线共享。例如用 4 片 1K×4bit 芯片组成 1K×16bit 的存储器
- **字扩展**：增加存储单元的数量（扩大地址空间）。用高位地址通过地址译码器选择芯片（片选信号 CS）。例如用 4 片 1K×8bit 芯片组成 4K×8bit 的存储器
- **字位同时扩展**：两者结合，同时增加字长和地址空间

## 四、指令系统

指令系统（也称指令集架构, ISA）是硬件和软件的接口——它定义了 CPU 能执行哪些操作、操作数在哪里、如何编码指令。ISA 是计算机体系结构中最重要的抽象层。

### 4.1 指令格式

一条指令由**操作码 (Opcode)** 和**地址码 (Operand)** 两部分组成。操作码指定要执行的操作，地址码指定操作数的位置。

| 指令类型 | 地址码个数 | 示例 | 说明 |
|----------|-----------|------|------|
| **零地址指令** | 0 | `NOP`, `HALT`, `PUSH`, `POP` | 无需操作数，或操作数隐含在栈中 |
| **一地址指令** | 1 | `INC R1`, `NOT R1` | 单操作数，结果存回原处 |
| **二地址指令** | 2 | `ADD R1, R2` | 最常见，R1 = R1 + R2 |
| **三地址指令** | 3 | `ADD R1, R2, R3` | R1 = R2 + R3，RISC 常用 |

### 4.2 寻址方式

寻址方式决定了如何从指令中找到操作数的实际地址（有效地址 EA, Effective Address）。

| 寻址方式 | 有效地址 EA | 特点 | 典型用途 |
|----------|------------|------|----------|
| **立即寻址** | 操作数直接在指令中 | 最快，但操作数大小受指令字长限制 | 常量赋值 `MOV R1, #5` |
| **直接寻址** | EA = 指令中给出的地址 A | 简单，一次内存访问 | 访问全局变量 |
| **间接寻址** | EA = (A)，即 A 指向的内存单元的内容是真正的地址 | 可扩大寻址范围，但需要两次内存访问 | 指针操作 |
| **寄存器寻址** | EA = 寄存器 R_i 的内容 | **最快**（不访问内存） | 频繁使用的变量 |
| **寄存器间接寻址** | EA = (R_i)，即寄存器中存放的是地址 | 较快，一次内存访问 | 通过指针访问数据 |
| **基址寻址** | EA = (BR) + A，基址寄存器 + 偏移量 | 支持程序**重定位**（改变 BR 即可） | 操作系统加载程序到不同地址 |
| **变址寻址** | EA = (IX) + A，变址寄存器 + 基地址 | 适合**数组访问**（IX 作为下标递增） | `a[i]` 的访问 |
| **相对寻址** | EA = (PC) + A，程序计数器 + 偏移量 | 指令地址与 PC 相关，代码位置无关 | **转移指令**（跳转到前后 A 条指令处） |
| **堆栈寻址** | EA = 栈顶 | 隐式操作栈顶 | 表达式求值、函数调用 |

### 4.3 CISC vs RISC

这是指令集设计的两大哲学。CISC 试图用复杂的指令简化编程，RISC 则用简单的指令追求硬件效率。

| 对比 | CISC (复杂指令集) | RISC (精简指令集) |
|------|------|------|
| 指令数量 | 多（>200 条） | 少（<100 条） |
| 指令长度 | 可变长（1-15 字节） | **定长**（通常 4 字节） |
| 寻址方式 | 多种（>10 种） | 少（通常 3-4 种） |
| 内存访问 | 任何指令都可以访问内存 | 只有 **Load/Store** 指令可以访问内存 |
| 执行周期 | 不同指令差异大（1-100+ 周期） | 大部分指令**单周期**完成 |
| 流水线 | 难以实现高效流水线 | **易于流水线化** |
| 寄存器 | 少（x86 最初只有 8 个通用寄存器） | 多（通常 32 个） |
| 代表 | x86 (Intel/AMD) | ARM, MIPS, RISC-V |

现代实际中，x86 处理器内部将 CISC 指令翻译为类 RISC 的微操作 (μops) 执行，兼取两者之长。

## 五、CPU

### 5.1 指令周期

CPU 执行一条指令的完整过程称为一个指令周期，包含若干机器周期（也称 CPU 周期）。

指令周期 = 取指周期 + [间址周期] + 执行周期 + [中断周期]

- **取指周期**：从内存中取出指令，放入指令寄存器 IR，PC 自增
- **间址周期**（可选）：如果使用间接寻址，需要额外访存获取有效地址
- **执行周期**：执行指令规定的操作
- **中断周期**（可选）：如果有中断请求，保存现场并转入中断处理

### 5.2 数据通路

数据通路是 CPU 内部数据流动的路径，包括寄存器、ALU、多路选择器以及连接它们的总线或专用线路。

- **CPU 内部数据通路**：寄存器之间的数据传送、寄存器到 ALU 的数据传送
- **CPU 与主存之间**：通过 **MAR (Memory Address Register)** 传送地址，通过 **MDR (Memory Data Register)** 传送数据

### 5.3 控制器

控制器是 CPU 的"指挥中心"，根据指令的操作码产生一系列控制信号，驱动数据通路完成指令的执行。

**硬布线控制器**：
- 用组合逻辑电路（门电路）根据指令操作码、时序信号、状态条件等直接产生控制信号
- 优点：速度快（纯硬件逻辑）
- 缺点：设计复杂，修改困难（改一条指令可能要重新布线），适合 RISC

**微程序控制器**：
- 将每条机器指令的控制信号序列存储在一个专用的只读存储器——**控制存储器 (CM, Control Memory)** 中
- 每条机器指令对应一段**微程序**，微程序由若干**微指令**组成，每条微指令产生一组**微操作**控制信号
- 优点：灵活，易于修改和扩展（改微程序而非改硬件），适合 CISC
- 缺点：速度比硬布线稍慢（需要从 CM 中读取微指令）

### 5.4 流水线 (Pipeline)

**基本思想**：将指令执行分为多个阶段，让不同指令的不同阶段同时执行——就像工厂流水线上同时处理多个产品的不同工序。

**经典五级流水线**：

| 阶段 | 缩写 | 功能 |
|------|------|------|
| 取指 | **IF** (Instruction Fetch) | 从内存取指令 |
| 译码 | **ID** (Instruction Decode) | 译码指令 + 读寄存器 |
| 执行 | **EX** (Execute) | ALU 运算 |
| 访存 | **MEM** (Memory Access) | 读写内存（Load/Store 指令） |
| 写回 | **WB** (Write Back) | 将结果写入目标寄存器 |

**理想加速比**：k 级流水线的理想加速比为 k（5 级流水线理想上比非流水线快 5 倍）。但实际受到**流水线冲突 (Hazard)** 的制约。

**三种流水线冲突**：

| 冲突类型 | 原因 | 解决方法 |
|----------|------|----------|
| **结构冲突** | 两条指令同时需要同一硬件资源。例如取指和访存同时需要访问内存 | 资源复制（如将 Cache 分为指令 Cache 和数据 Cache）；或插入气泡等待 |
| **数据冲突** | 后续指令需要使用前面指令还未产生的结果 | **转发/旁路 (Forwarding/Bypassing)**：EX 阶段产生的结果直接转发给下一条指令的 EX 输入；插入气泡 (Stall)；编译器指令调度 |
| **控制冲突** | 分支指令改变了 PC，但后续指令已经被取入流水线 | 分支预测、延迟分支（在分支后放一条与分支无关的指令）、分支目标缓冲 (BTB) |

**数据冲突的三种类型**：

| 类型 | 含义 | 发生条件 | 频率 |
|------|------|----------|------|
| **RAW (Read After Write)** | 读后写，真数据依赖 | 后面的指令要读前面指令写的结果 | **最常见** |
| **WAR (Write After Read)** | 写后读，反依赖 | 后面的指令要写前面指令读的寄存器 | 乱序执行时可能出现 |
| **WAW (Write After Write)** | 写后写，输出依赖 | 两条指令写同一个寄存器 | 乱序执行时可能出现 |

**分支预测 (Branch Prediction)**：

- **静态预测**：
  - 总是预测不跳转（对顺序代码有效）
  - 总是预测跳转（对循环有效——循环末尾的分支大部分时候都跳回循环体）
  - 根据分支方向预测：向后跳转（循环）预测跳转，向前跳转预测不跳转

- **动态预测**：基于运行时历史信息
  - **1-bit 预测器**：记录上次分支是否跳转，预测这次也一样
  - **2-bit 饱和计数器**：需要连续两次预测错误才改变预测方向（比 1-bit 更稳定，对循环效果好——循环最后一次退出时预测错误，但不会影响下一次循环的预测）
  - **分支历史表 (BHT, Branch History Table)**：为多个分支指令维护各自的预测器

现代 CPU 的分支预测准确率可达 95% 以上。

## 六、总线

总线 (Bus) 是连接计算机各功能部件的公共通信线路。多个部件共享总线进行信息传输。

### 6.1 总线分类

**按功能分**：
- **数据总线 (Data Bus)**：传输数据，**双向**。总线宽度决定一次传输的数据量（如 64 位数据总线一次传 8 字节）
- **地址总线 (Address Bus)**：传输地址，**单向**（CPU → 内存/I/O）。宽度决定可寻址空间（32 位地址总线 → 4GB 地址空间）
- **控制总线 (Control Bus)**：传输控制信号（读/写、中断、总线请求等），部分单向部分双向

**按层次分**：
- **片内总线**：CPU 内部，连接寄存器和 ALU
- **系统总线**：CPU、主存、I/O 接口之间的总线
- **通信总线/I/O 总线**：连接外设（如 PCIe、USB、SATA）

### 6.2 总线仲裁

多个设备可能同时请求使用总线，需要仲裁机制决定谁先使用。

- **集中仲裁**：由总线控制器统一决定
  - **链式查询（菊花链）**：总线授权信号沿链传递，离控制器近的设备优先级高。简单但不公平
  - **计数器定时查询**：控制器用计数器轮询设备。可以设定起始值来调整优先级
  - **独立请求**：每个设备有独立的请求和授权线。最快但线数最多
- **分布仲裁**：各设备自行判断（如通过比较优先级编号）

### 6.3 总线传输

- **同步传输**：所有操作由统一时钟控制。简单但受限于最慢的设备
- **异步传输**：通过请求-应答握手协议。灵活但控制复杂。分为不互锁、半互锁、全互锁三种
- **半同步传输**：同步 + WAIT 信号。慢设备通过拉低 WAIT 信号请求等待

**总线带宽 (Bandwidth)** = 总线宽度 × 总线工作频率。例如 64 位宽、800MHz 的总线带宽 = 64 × 800M / 8 = 6.4 GB/s。

## 七、I/O 系统

### 7.1 I/O 接口

I/O 接口（也称设备控制器/适配器）是 CPU 与外部设备之间的"翻译官"，负责速度匹配、数据格式转换和信号电平转换。

I/O 接口包含：
- **数据缓冲寄存器**：暂存传输的数据
- **状态/控制寄存器**：设备状态信息和 CPU 发出的控制命令
- **地址译码逻辑**：识别 CPU 发出的 I/O 地址

**编址方式**：
- **统一编址（内存映射 I/O, Memory-Mapped I/O）**：I/O 端口和内存共用同一个地址空间。用普通的 Load/Store 指令访问 I/O。简单统一，ARM 采用
- **独立编址（I/O 映射 I/O, Port-Mapped I/O）**：I/O 端口有独立的地址空间，需要专门的 I/O 指令（如 x86 的 IN/OUT）。地址空间不冲突但需要额外的指令

### 7.2 中断系统

**中断 (Interrupt)** 是 CPU 暂停当前程序，转去处理紧急事件的机制。中断使得 CPU 不必轮询设备状态，可以在等待 I/O 的同时执行其他程序。

**中断处理过程**：
1. **中断请求**：外设完成操作后通过中断请求线发出信号
2. **中断响应**：CPU 在**当前指令执行完毕后**（不是执行过程中）检查中断请求
3. **中断处理**：
   - **保护现场**：将 PC、PSW（程序状态字）、关键寄存器压栈
   - 通过**中断向量**（中断服务程序的入口地址）跳转到中断服务程序
   - 执行中断服务程序
   - **恢复现场**：从栈中恢复寄存器
   - **中断返回**：执行 IRET 指令，恢复 PC 和 PSW，继续被中断的程序

**中断向量表 (Interrupt Vector Table)**：存放各中断源对应的中断服务程序入口地址的数组。CPU 根据中断类型号查表获得跳转地址。

**中断优先级**：不同中断源的紧急程度不同。通常：硬件故障 > I/O 中断 > 程序中断 > 外部中断

**中断嵌套**：高优先级的中断可以打断正在处理的低优先级中断。通过**中断屏蔽字**实现——每个中断源对应一个屏蔽位，1 表示屏蔽（禁止该中断），0 表示允许。通过设置不同中断服务程序的屏蔽字，可以灵活控制中断嵌套的行为。

### 7.3 I/O 方式

| 方式 | 原理 | CPU 参与度 | 适用场景 |
|------|------|-----------|----------|
| **程序查询** | CPU 不断轮询设备状态寄存器 | 极高（CPU 忙等） | 简单低速设备 |
| **中断驱动** | 设备就绪后发中断通知 CPU | 中（每次传输仍需 CPU 搬运数据） | 中速设备 |
| **DMA (Direct Memory Access)** | DMA 控制器直接在设备与内存间传输数据，CPU 仅在开始和结束时参与 | 低 | 高速块设备（磁盘、网卡） |
| **通道** | 独立的 I/O 处理器，有自己的指令集 | 极低 | 大型机、高吞吐 I/O |

## 八、性能评估

### 8.1 CPU 性能公式 (Iron Law)

CPU 执行时间由三个因素决定：

```
CPU Time = Instruction Count × CPI × Clock Cycle Time
         = IC × CPI × (1 / Clock Rate)
```

- **IC (Instruction Count)**：程序执行的指令总数，取决于 ISA 和编译器
- **CPI (Cycles Per Instruction)**：每条指令平均需要的时钟周期数，取决于微架构（Organization）和 ISA
- **Clock Cycle Time**：单个时钟周期的时间，取决于硬件工艺（Technology）和微架构

**IPC (Instructions Per Cycle)** = 1 / CPI，现代超标量处理器的 IPC 通常 > 1。

**有效 CPI 计算**：不同类型指令有不同的 CPI，总体有效 CPI 为加权平均：

```
CPI_eff = Σ (CPI_i × Freq_i)
```

其中 Freq_i 是第 i 类指令在程序中的占比。

**举例**：假设 ALU 占 50%（CPI=1）、Load 占 20%（CPI=5）、Store 占 10%（CPI=3）、Branch 占 20%（CPI=2）：
CPI_eff = 0.5×1 + 0.2×5 + 0.1×3 + 0.2×2 = 2.2

### 8.2 Amdahl's Law

**Amdahl 定律**量化了部分加速对整体性能的影响：

```
Speedup = 1 / ((1 - P) + P / S)
```

- P：可被加速的部分占总执行时间的比例
- S：该部分的加速比

**核心启示**：
- **Make the Common Case Fast**：应当优化占执行时间比例大的部分
- **收益递减**：即使将某部分加速到无穷大，整体加速比上限为 1/(1-P)
- 例：某操作占总时间 83%，即使加速 2 倍，总体也只快 11.1%（如果剩余 17% 不可优化）

### 8.3 性能比较与汇总

- **Latency（延迟）**：完成一个任务需要多长时间（对用户重要）
- **Throughput（吞吐量）**：单位时间完成多少任务（对数据中心重要）
- "X is n times faster than Y" 意味着 Performance_X / Performance_Y = Time_Y / Time_X = n
- 多程序性能汇总时使用**几何平均数 (Geometric Mean)**，而非算术平均——几何平均对参考机器的选择不敏感
- 比较 CPI 时须用**调和平均数 (Harmonic Mean)**，因为 IPC = 1/CPI 是倒数关系

## 九、RISC-V 指令集架构

RISC-V 是一个**开源的 RISC 指令集架构**，起源于 UC Berkeley（2010 年），采用模块化设计。

### 9.1 设计原则

- **简单性优先规律性**：固定长度指令（32 位）、少量指令格式
- **好的设计需要好的折衷**：6 种指令格式
- **越小越快**：有限的指令集、32 个通用寄存器、少量寻址方式
- **加速常见情况**：算术操作数来自寄存器（Load/Store 架构）、支持立即数操作

### 9.2 寄存器约定

RISC-V 有 32 个 64 位通用寄存器（x0-x31）：

| 寄存器 | ABI 名称 | 用途 | 调用保存 |
|--------|---------|------|----------|
| x0 | zero | 硬连线为常数 0 | N/A |
| x1 | ra | 返回地址 | Caller |
| x2 | sp | 栈指针 | Callee |
| x5-x7 | t0-t2 | 临时寄存器 | Caller |
| x8-x9 | s0-s1 | 保存寄存器/帧指针 | Callee |
| x10-x17 | a0-a7 | 函数参数/返回值 | Caller |
| x18-x27 | s2-s11 | 保存寄存器 | Callee |
| x28-x31 | t3-t6 | 临时寄存器 | Caller |

**Caller-saved**：调用者在调用前负责保存（被调用者可自由使用）
**Callee-saved**：被调用者在使用前负责保存（调用者期望返回后不变）

### 9.3 指令格式

RISC-V 有 6 种基本指令格式，均为 32 位宽：

```
R 格式: | funct7(7) | rs2(5) | rs1(5) | funct3(3) | rd(5)  | opcode(7) |  -- 寄存器-寄存器运算
I 格式: | imm[11:0](12)       | rs1(5) | funct3(3) | rd(5)  | opcode(7) |  -- 立即数运算/Load
S 格式: | imm[11:5](7)| rs2(5) | rs1(5) | funct3(3) | imm[4:0](5) | opcode(7) |  -- Store
SB格式: | imm[12,10:5]| rs2(5) | rs1(5) | funct3(3) | imm[4:1,11] | opcode(7) |  -- 条件分支
U 格式: | imm[31:12](20)                             | rd(5)  | opcode(7) |  -- 高位立即数
UJ格式: | imm[20,10:1,11,19:12](20)                  | rd(5)  | opcode(7) |  -- 无条件跳转
```

**关键指令示例**：

| 类别 | 指令 | 格式 | 含义 |
|------|------|------|------|
| 算术 | `add x5, x6, x7` | R | x5 = x6 + x7 |
| 立即数 | `addi x5, x6, 20` | I | x5 = x6 + 20 |
| 加载 | `ld x5, 40(x6)` | I | x5 = Memory[x6+40] |
| 存储 | `sd x5, 40(x6)` | S | Memory[x6+40] = x5 |
| 条件分支 | `beq x5, x6, L` | SB | if(x5==x6) goto PC+L |
| 跳转链接 | `jal x1, Proc` | UJ | x1=PC+4; goto Proc |
| 高位加载 | `lui x5, imm` | U | x5 = imm << 12 |

### 9.4 RISC-V 寻址方式

RISC-V 只有 4 种寻址方式（体现 RISC 简洁设计）：

1. **寄存器寻址**：操作数在寄存器中（R 格式）
2. **基址偏移寻址**：EA = 寄存器内容 + 12 位偏移量（I/S 格式，用于 Load/Store）
3. **立即数寻址**：操作数嵌入指令中的 12 位立即数（I 格式）
4. **PC 相对寻址**：分支目标 = PC + 偏移量（SB/UJ 格式）

### 9.5 原子操作与同步

RISC-V 用两条 R 型指令实现原子操作，替代 CISC 的 Compare-and-Swap：

- **`lr.d` (Load-Reserved)**：从内存加载数据并设置预留标记
- **`sc.d` (Store-Conditional)**：仅当预留标记仍有效时写入内存，成功返回 0，失败返回 1

```assembly
retry:  lr.d  x5, (x6)       # x5 = Memory[x6]，设预留
        # ... 修改 x5 ...
        sc.d  x7, x5, (x6)   # 条件存储
        bnez  x7, retry       # 失败则重试
```

## 十、高级流水线与指令级并行 (ILP)

### 10.1 超流水线 (Super-pipelining)

将流水线各阶段进一步细分为多个子阶段，从而提高时钟频率。

- 若将每个阶段分为 m 个子阶段，时钟周期缩短为 T/m
- **理想情况**：每 T/m 产出一条指令结果
- **问题**：流水线越深，分支预测错误和数据依赖的惩罚越大（需清洗更多阶段）
- **存在最优深度**：约 15 级左右，超过后收益递减（Pentium 4 曾达到 31 级，效果不佳）

### 10.2 超标量 (Superscalar)

**动态多发射处理器**：硬件在运行时决定每个周期发射多条指令。

- N-way Superscalar：每周期最多发射 N 条指令，需要 N 套功能单元
- 硬件负责检测指令间依赖关系
- **理想加速比**：N（所有指令独立时），实际受限于依赖关系
- 超标量与超流水线正交——可以同时使用
- 代表：Intel Core 系列、IBM Power、ARM Cortex-A

**超标量的挑战**：
- 需要同时检查多条指令间的 RAW/WAR/WAW 依赖
- 需要检查正在解码的指令与流水线中已有指令的冲突
- 硬件复杂度随发射宽度急剧增加

### 10.3 VLIW (Very Long Instruction Word)

**静态多发射处理器**：编译器在编译时决定并行执行的指令。

- 一条 VLIW 指令包含多个"子指令"，分别对应不同功能单元
- 子指令之间**必须独立**（由编译器保证）
- 硬件**不需要做依赖检查**，比超标量简单

| 对比 | Superscalar | VLIW |
|------|-------------|------|
| 并行决策者 | 硬件（运行时） | 编译器（编译时） |
| 硬件复杂度 | 高 | 低 |
| 代码兼容性 | 好（ISA 不变） | 差（换硬件需重编译） |
| 代码膨胀 | 无 | 有（NOP 填充未用 slot） |
| 代表 | Intel Core, ARM | Intel Itanium (IA-64) |

### 10.4 编译器优化技术

**寄存器重命名 (Register Renaming)**：消除 WAR 和 WAW 假依赖——将不同指令写同一寄存器的情况映射到不同物理寄存器。硬件或编译器都可以做。

**循环展开 (Loop Unrolling)**：将 M 次迭代的循环展开为 M/N 次迭代（每次执行 N 次循环体）：
- 减少循环控制开销（分支指令减少）
- 暴露更多 ILP（跨迭代的指令可并行调度）
- 代价：代码膨胀

**函数内联 (Function Inlining)**：将被调用函数体嵌入调用处，消除 CALL/RET 开销。

**树高度缩减 (Tree Height Reduction)**：利用运算结合律缩短关键路径。如 `((a+b)+c)+d` 改为 `(a+b)+(c+d)`，从 3 级依赖变为 2 级。

**软件流水线 (Software Pipelining)**：将不同迭代的不同阶段重叠执行，仅在循环开始和结束有 prolog/epilog 开销，而非每次迭代都有。

### 10.5 分支预测深入

**Branch Target Buffer (BTB)**：缓存分支指令地址及其目标地址的硬件结构。在 IF 阶段用 PC 查 BTB，命中则直接获取目标地址，无需等待译码。

**Return Address Stack (RAS)**：专门预测函数返回地址的硬件栈。CALL 时压入返回地址，RET 时弹出预测地址。8-16 项即可达到很高准确率。

**高级分支预测**：
- **gshare**：将分支地址与全局历史寄存器异或后索引 PHT（Pattern History Table），减少 aliasing
- **Tournament/Hybrid Predictor**：同时使用多个预测器，用一个选择器选择更准确的（如 Alpha 21264）
- **Perceptron Predictor**：用感知器神经网络替代 2-bit 饱和计数器，可利用更长的历史信息，硬件资源线性增长

## 十一、动态调度

### 11.1 乱序执行 (Out-of-Order, OoO)

**核心思想**：硬件在运行时重排指令执行顺序，在保证程序正确性的前提下减少流水线停顿。

**OoO 的性能优势**：
- In-Order 处理器遇到 cache miss 等长延迟操作时后续指令全部停顿
- OoO 处理器可以跳过等待中的指令，先执行后面不依赖的指令
- 2-wide OoO 可以比 2-wide In-Order 快显著提升

**OoO 的问题**：必须正确处理三种数据依赖（RAW、WAR、WAW）

### 11.2 Scoreboard 记分牌

**CDC 6600 (1964)** 首次实现的动态调度机制，通过**记分牌**跟踪指令状态、功能单元状态和寄存器状态。

**流水线阶段拆分**：将 Decode 阶段拆分为三个子阶段：
1. **Issue (发射)**：按程序顺序分配功能单元，检查结构冲突和 WAW
2. **Read Operands (读操作数)**：等待源操作数就绪（解决 RAW）
3. **Execute (执行)**：在功能单元上执行
4. **Write Back (写回)**：等待无 WAR 冲突后写回结果

**记分牌三张表**：
- **指令状态表**：跟踪每条指令处于哪个阶段
- **功能单元状态表**：每个 FU 的忙闲、操作类型、目标/源寄存器、源寄存器就绪标志
- **寄存器结果状态表**：每个寄存器将由哪个 FU 写入

**Scoreboard 的局限**：
- 不做寄存器重命名，WAR 和 WAW 仍会导致停顿
- Issue 阶段顺序发射，一旦遇到结构冲突或 WAW 就阻塞后续指令

### 11.3 Tomasulo 算法

IBM 360/91 (1967) 提出的更先进的动态调度算法，关键创新是**保留站 (Reservation Station)** 和**硬件寄存器重命名**。

**核心机制**：
- **保留站 (RS)**：每个功能单元前有若干保留站，缓冲等待执行的指令及其操作数
- **公共数据总线 (CDB, Common Data Bus)**：功能单元计算完成后将结果广播到 CDB，所有等待该结果的保留站同时获取
- **隐式寄存器重命名**：保留站编号替代寄存器编号，自动消除 WAR 和 WAW

**Tomasulo 三阶段**：
1. **Issue**：从指令队列取指令，分配保留站，读取已就绪的操作数，将保留站编号作为重命名标记
2. **Execute**：操作数全部就绪后开始执行
3. **Write Result**：通过 CDB 广播结果，更新所有等待该结果的保留站和寄存器文件

**对比 Scoreboard**：
- Tomasulo 通过保留站实现了**分布式**的冲突检测
- 消除了 WAR 和 WAW 依赖（Scoreboard 无法消除）
- 允许多条同类型指令同时等待（不受功能单元数限制）

### 11.4 Reorder Buffer (ROB) 重排序缓冲

**问题**：Tomasulo 的乱序完成可能导致精确异常 (Precise Exception) 无法实现——如果已完成的指令后来需要被取消（分支预测错误或异常），无法恢复。

**ROB 的作用**：在 Tomasulo 基础上增加一个按程序顺序的环形缓冲区，指令按序进入 ROB，乱序执行，但**按序提交 (Commit)**。

**带 ROB 的 Tomasulo 四阶段**：
1. **Issue**：分配保留站 + ROB 条目
2. **Execute**：操作数就绪后执行
3. **Write Result**：结果写入 ROB 条目（而非直接写寄存器）
4. **Commit**：ROB 头部的指令顺序提交——将结果从 ROB 写入寄存器/内存

**精确异常保证**：异常发生时，ROB 中该指令之前的已提交，之后的全部丢弃，处理器状态回到异常指令之前的精确状态。

## 十二、Cache 性能分析与优化

### 12.1 Cache 性能度量

**AMAT (Average Memory Access Time)**：

```
AMAT = Hit Time + Miss Rate × Miss Penalty
```

**包含 Cache 的 CPU 时间**：

```
CPU Time = IC × (CPI_ideal + Cycles_memory_stall) × Clock Cycle
Cycles_memory_stall = I$ miss rate × penalty + Data_access_rate × D$ miss rate × penalty
```

**举例**：CPI_ideal=2，Miss Penalty=100 cycles，I$ miss rate=2%，D$ miss rate=4%，36% 指令为 LD/ST：
- Memory stall = 2%×100 + 36%×4%×100 = 3.44 cycles
- CPI_stall = 2 + 3.44 = 5.44
- 内存停顿占总时间 63%

**关键观察**：随着处理器速度提升（CPI_ideal 降低或频率翻倍），Cache miss 的相对惩罚反而**增大**（Memory Wall 问题）。

### 12.2 Cache 失效分类 (3C Model)

| 类型 | 英文 | 原因 | 解决方法 |
|------|------|------|----------|
| **强制失效** | Compulsory (Cold) | 首次访问某块，无法避免 | 增大块大小；预取 (Prefetching) |
| **容量失效** | Capacity | Cache 容量不足以容纳工作集 | 增大 Cache 容量 |
| **冲突失效** | Conflict | 多个块映射到同一组，竞争导致替换 | 提高相联度；Victim Cache |

### 12.3 Pseudo-LRU 替换

对于 N-way 组相联（N>2），精确 LRU 开销大。实际使用**Pseudo-LRU (PLRU)**：用二叉树结构近似 LRU。

- N-way Cache 的每组用 N-1 位构建二叉树
- 每个内部节点的位表示"左子树更旧(0)"或"右子树更旧(1)"
- 访问某路时，沿路径将所有节点指向远离该路的方向
- 替换时，沿着"更旧"方向走到叶子节点，即为替换目标

**Clock 算法**：更简单的近似 LRU——用循环指针和引用位(R)。R=0 替换该页；R=1 置 R=0 继续扫描。

### 12.4 Cache 优化技术详解

目标：减小 AMAT = Hit Time + Miss Rate × Miss Penalty

**降低 Miss Penalty**：

| 技术 | 原理 |
|------|------|
| **多级 Cache** | L1→L2→L3→主存，逐级增大容量和延迟。包含/互斥/非包含三种层次关系 |
| **Critical Word First** | 发生 miss 时，优先从内存请求 CPU 需要的那个字，到达后立即送 CPU |
| **Early Restart** | 请求字到达后立即重启 CPU，不等整个块加载完 |
| **Write Combining Buffer** | 合并多次写操作后统一写回，减少总线事务 |
| **Non-blocking Cache (MSHR)** | 允许多个 outstanding miss，用 MSHR (Miss Status Holding Register) 跟踪。CPU 不必在一次 miss 时完全停顿 |

**降低 Miss Rate**：

| 技术 | 原理 |
|------|------|
| **增大块大小** | 利用空间局部性，但太大则减少 Cache 行数（增加冲突） |
| **提高相联度** | 从直接映射到 2-way 的收益最大 |
| **硬件预取 (Prefetching)** | 硬件检测访存模式，提前将数据加载到 Cache。流式预取器 (Stream Buffer) 预取连续块 |
| **编译器优化** | Loop Interchange（改善空间局部性）、Loop Fusion（改善时间局部性）、Merging Arrays（合并数组改善空间局部性） |

**降低 Hit Time**：

| 技术 | 原理 |
|------|------|
| **小而简单的 Cache** | L1 Cache 通常较小、低相联度以保证快速访问 |
| **Virtually Indexed Cache** | 用虚拟地址索引 Cache，避免等待地址翻译。需解决别名问题 |
| **Way Prediction** | 预测命中在哪一路，先访问该路，预测正确则 hit time 接近直接映射 |
| **Trace Cache** | 缓存已解码的指令序列（跨基本块），减少取指开销（Pentium 4 采用） |

## 十三、主存 DRAM 详解

### 13.1 DRAM 子系统层次

```
Channel（通道）
  └── DIMM（内存条）
        └── Rank（列）-- 多个 Chip 组成，提供完整总线宽度
              └── Chip（芯片）
                    └── Bank（存储体）-- 可独立操作
                          └── MAT（子阵列）
                                └── Row × Column 的 DRAM 单元阵列
```

- **MAT**：基本存储阵列，包含行译码器、列译码器和感应放大器（Row Buffer）
- **Bank**：多个 MAT 组成，是最小的独立操作单位。每个 Bank 有自己的 Row Buffer
- **Chip**：多个 Bank + 外围逻辑和 I/O 接口
- **Rank**：多个 Chip 并联以提供完整数据总线宽度（如 8 个 ×8 芯片组成 64 位 Rank）
- **Channel**：一套独立的数据/地址/命令总线

### 13.2 DRAM 操作与时序

DRAM 访问分三步：

1. **Activate (行激活)**：打开一行，将数据读入 Row Buffer → 耗时 **tRCD**
2. **Read/Write (列访问)**：从 Row Buffer 中选取列数据 → 耗时 **tCAS (CL)**
3. **Precharge (预充电)**：关闭当前行，准备下一次行访问 → 耗时 **tRP**

**关键时序参数**：

| 参数 | 含义 |
|------|------|
| **tRCD** | Row-Column Delay，行激活到列访问的最小间隔 |
| **tCAS (CL)** | Column Access Strobe Latency，列读命令到数据输出的延迟 |
| **tRP** | Row Precharge，预充电所需时间 |
| **tRAS** | Row Access Strobe，行激活到预充电的最小间隔 |
| **tRC** | Row Cycle = tRAS + tRP，同一 Bank 两次行访问的最小间隔 |
| **tCCD** | Column-to-Column Delay，连续列访问间隔 |

**Row Buffer 策略**：
- **Open-page policy**：行保持打开，利用局部性。Row Buffer Hit 很快，但 Row Buffer Miss（Bank Conflict）需要先 Precharge 再 Activate
- **Close-page policy**：访问后立即 Precharge。适合低局部性的访问模式

### 13.3 DRAM 的挑战

- **破坏性读取 (Destructive Read)**：读操作会破坏电容中存储的数据。读后必须回写（Restore），由感应放大器完成
- **电荷泄漏**：即使不读取，电容也会慢慢漏电（这是"Dynamic"的含义）。必须**周期性刷新 (Refresh)**，通常每 64ms 刷新所有行

### 13.4 DDR 技术演进

**DDR (Double Data Rate)**：在时钟信号的上升沿和下降沿都传输数据，有效传输速率翻倍。

DDR 代际演进的核心思路：**Prefetch（预取）** —— 保持内核频率不变，通过增加内部总线宽度与外部总线的比值来提升传输速率。

| DDR 版本 | Prefetch 比 | 典型传输速率 | 核心频率 | 电压 |
|----------|-------------|-------------|---------|------|
| DDR1 | 2n (2:1) | 200-400 MT/s | 100-200 MHz | 2.5V |
| DDR2 | 4n (4:1) | 400-800 MT/s | 100-200 MHz | 1.8V |
| DDR3 | 8n (8:1) | 800-2133 MT/s | 100-266 MHz | 1.5V |
| DDR4 | 8n (8:1) | 1600-3200 MT/s | 200-400 MHz | 1.2V |
| DDR5 | 16n (16:1) | 3200-6400 MT/s | 200-400 MHz | 1.1V |

**高速信号挑战与解决方案**：
- **时钟偏斜 (Clock Skew)**：用源同步 (Source-Synchronous) 技术，发送端附带数据选通信号 (DQS)
- **信号反射**：在 DIMM 末端或芯片上加终端电阻 (Termination) 吸收反射信号
- **信号拓扑**：DDR3 采用 Fly-by 拓扑（串联），信号延迟更可预测，便于校正

### 13.5 内存控制器与调度

**内存控制器 (Memory Controller)** 位于 CPU 与 DRAM 之间，负责：
1. 缓冲 CPU 请求 → 2. 地址翻译（物理地址→Bank/Row/Column） → 3. 命令调度 → 4. 发送 DRAM 命令 → 5. 遵守时序约束

**地址映射策略**（物理地址到 DRAM 地址的映射方式）：
- **RaBaRoCo (Row-Bank-Row-Column)**：不同映射方式影响 Row Buffer 命中率和 Bank 级并行度
- 目标：行地址使用低熵（高局部性）的位，Bank 地址使用高熵（高变化性）的位

**Bank 级并行**：不同 Bank 可以独立操作。当一个 Bank 在 Activate/Precharge 时，另一个 Bank 可以传输数据。

**NUMA (Non-Uniform Memory Access)**：多 CPU 系统中，每个 CPU 有本地内存。访问本地内存快，访问远端 CPU 的内存慢（需要通过互连，如 Intel UPI）。

## 十四、多处理器与 Cache 一致性

### 14.1 Flynn 分类法

按指令流和数据流的单/多分为四类：

| 分类 | 含义 | 代表 |
|------|------|------|
| **SISD** | 单指令单数据 | 传统单核处理器 |
| **SIMD** | 单指令多数据 | GPU、向量处理器、SSE/AVX |
| **MISD** | 多指令单数据 | 极少见（实验性系统） |
| **MIMD** | 多指令多数据 | **现代多核处理器**、超级计算机 |

### 14.2 存储一致性 vs Cache 一致性

| 概念 | Memory Consistency（存储一致性） | Cache Coherence（缓存一致性） |
|------|------|------|
| 关注范围 | 所有处理器对**所有地址**的访问顺序 | 所有处理器对**同一地址**的访问顺序 |
| 本质 | **全局排序**问题 | **局部排序**问题 |
| 影响 | 程序员可见的内存行为模型 | 硬件透明地维护 |

### 14.3 存储一致性模型

**Sequential Consistency (SC，顺序一致性)**：
- 所有处理器的内存操作看起来像按某个**全局顺序**交错执行
- 每个处理器内部的操作按**程序顺序**出现在全局顺序中
- 最强、最直观的一致性模型，但限制了硬件优化空间

**Total Store Order (TSO)**：
- 最广泛使用的一致性模型（x86 采用）
- 与 SC 的区别：每个核心有一个 FIFO 写缓冲区 (Write Buffer)
- **Store-Load 可以乱序**（Store 可以延迟到 Write Buffer 中，后续 Load 先执行），其余顺序保持
- 需要 **FENCE 指令**在关键位置显式强制顺序

**更弱的模型**：
- **PSO (Partial Store Order)**：Store-Store 也可以乱序
- **Relaxed Memory Order**：几乎所有操作都可以乱序，依赖 fence 保证关键顺序
- **Weak Consistency**：仅在同步点（barrier）保证一致性

### 14.4 Cache 一致性协议

**Write-Through Cache 一致性**：
- 写时同时更新主存 → 主存始终最新
- 总线嗅探 (Bus Snooping)：其他 Cache 监听总线，发现有写操作命中自己的 Cache 行时**无效化 (Invalidate)** 该行
- 两个状态：Valid (V) / Invalid (I)

**MESI 协议**（Write-Back Cache 的标准一致性协议）：

每个 Cache 行有 4 种状态：

| 状态 | 含义 | 可读 | 可写 | 内存一致 | 共享 |
|------|------|------|------|---------|------|
| **M (Modified)** | 已修改，是唯一且最新的副本 | 是 | 是 | 否 | 否 |
| **E (Exclusive)** | 未修改，是唯一的副本 | 是 | 是(→M) | 是 | 否 |
| **S (Shared)** | 未修改，可能有其他副本 | 是 | 否(需BusRdX) | 是 | 是 |
| **I (Invalid)** | 无效 | 否 | 否 | - | - |

**总线事务**：
- **BusRd**：读 miss 时发出，请求数据
- **BusRdX**：写 miss 或写 Shared 行时发出，请求独占访问并使其他副本无效
- **Flush**：将数据放到总线上供请求者和内存获取

**典型状态转换示例**：
- Cold Miss (无其他副本)：I → E
- Cold Miss (有其他副本)：I → S（其他 E/M 的副本也变为 S）
- 本地写（E 状态）：E → M（无需总线事务）
- 本地写（S 状态）：S → M（需发 BusRdX 使其他副本无效化）
- 其他核心 BusRd（M 状态）：M → S（Flush 数据到总线和内存）
- 其他核心 BusRdX（任意有效状态）：→ I（被无效化）

### 14.5 MESI 协议变体

| 协议 | 增加的状态 | 主要改进 | 代表处理器 |
|------|----------|----------|-----------|
| **MSI** | 无 E 状态 | 更简单，但独占写需总线事务 | RISC-V 实现 |
| **MEI** | 无 S 状态 | 读 miss 时直接独占，减少共享开销 | PowerPC 755 |
| **MOESI** | +Owned (O) | M→O 避免回写内存（O 负责供数据） | AMD Opteron |
| **MESIF** | +Forward (F) | 多个 S 中选一个 F 代表回复请求，避免重复传输 | Intel Nehalem |

### 14.6 Snooping vs Directory 一致性

| 方式 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **总线嗅探 (Snooping)** | 所有 Cache 监听共享总线上的事务 | 实现简单、延迟低 | 总线带宽瓶颈，不可扩展 | 小规模多核（<16 核） |
| **目录协议 (Directory)** | 用一个目录记录每个内存块的共享者列表 | 可扩展到大规模系统 | 目录存储开销、间接访问延迟 | 大规模多核/多处理器 |

**目录协议基本操作**：
- 每个内存块在目录中有一组 presence bits（每个处理器一位）和一个 modified bit
- **Read Miss (Clean)**：请求者联系 Home 节点 → Home 发送数据 → 更新 presence bits
- **Read Miss (Dirty)**：Home 通知 Owner → Owner 将数据发给请求者和 Home → 更新状态
- **Write Miss**：Home 发送所有 sharers 列表 → 请求者向所有 sharers 发 invalidate → sharers 回复 ACK → 写操作完成

**分布式目录**：在 NUMA 系统中，每个节点维护本地内存块的目录信息，避免集中式目录成为瓶颈。

## 十五、GPU 体系结构简介

### 15.1 GPU vs CPU

| 对比 | CPU | GPU |
|------|-----|-----|
| 设计目标 | 低延迟、通用计算 | 高吞吐量、数据并行 |
| 核心数 | 少量强核心（4-64） | 大量简单核心（数千） |
| 控制逻辑 | 复杂（OoO、分支预测） | 简单（SIMT 执行模型） |
| 缓存 | 大容量多级 Cache | 较小 Cache，依赖线程切换隐藏延迟 |
| 适用场景 | 通用程序、低并行度 | 图形渲染、AI/ML、科学计算 |

### 15.2 GPU 编程模型 (CUDA)

- **Thread（线程）**：最小执行单位
- **Warp**：32 个线程组成一个 Warp，以 SIMT 方式执行同一条指令
- **Block (Thread Block)**：一组线程共享 Shared Memory，在同一个 SM 上执行
- **Grid**：所有 Block 的集合

### 15.3 图形渲染流水线

```
几何处理 → 光栅化 → 像素处理 → 合并输出
```

1. **几何处理**：顶点变换（Object→World→Eye→Clip 坐标系）、裁剪
2. **光栅化 (Rasterization)**：将三角形投影到屏幕像素网格，确定每个像素覆盖的图元
3. **像素处理**：纹理映射 (Texture)、光照计算（Gouraud/Phong Shading）
4. **合并输出**：深度测试 (Z-Buffer)、Alpha Blending（透明度混合）
