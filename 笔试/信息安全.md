# 信息安全

## 一、密码学基础

密码学是信息安全的基石。它的核心目标是保护信息的**机密性**（只有授权者能读取）、**完整性**（信息未被篡改）和**可用性**（授权者能正常使用信息）。密码学提供了一套数学工具，使得即使在不安全的通信信道上，也能安全地传输和存储信息。

### 1.1 对称加密 vs 非对称加密

加密是将明文（可读信息）转换为密文（不可读信息）的过程，解密则是逆过程。根据加密和解密是否使用同一密钥，分为对称加密和非对称加密两大类。

**对称加密**：加密和解密使用**同一把密钥**。就像一把锁只有一把钥匙，发送方和接收方必须事先共享这把密钥。优点是速度快（比非对称加密快数百倍），适合加密大量数据；缺点是密钥分发困难——如何在不安全的信道上安全地将密钥传给对方？如果有 n 个人需要两两通信，需要 C(n,2) = n(n-1)/2 把密钥，管理复杂度很高。

**非对称加密**：使用一对密钥——**公钥**和**私钥**。公钥可以公开给任何人，私钥只有持有者知道。用公钥加密的数据只能用对应的私钥解密，反之亦然。这解决了密钥分发的问题：任何人都可以用你的公钥加密消息发给你，只有你能用私钥解密。缺点是计算速度慢，不适合加密大量数据。

**实际应用中的混合加密**：HTTPS/TLS 等协议结合两者优势——用非对称加密安全地交换一个临时的对称密钥（会话密钥），然后用这个对称密钥加密后续的大量通信数据。

| 对比 | 对称加密 | 非对称加密 |
|------|----------|-----------|
| 密钥 | 加密解密使用同一密钥 | 公钥加密、私钥解密（或反向用于签名） |
| 速度 | 快（约百倍以上） | 慢 |
| 密钥分发 | 困难（需安全通道） | 容易（公钥可公开） |
| 密钥管理 | n 人通信需 n(n-1)/2 把密钥 | n 人通信只需 n 对密钥 |
| 适用 | 大量数据加密 | 密钥交换、数字签名、身份认证 |

### 1.2 对称加密算法

| 算法 | 密钥长度 | 分组大小 | 特点 |
|------|----------|----------|------|
| DES | 56 bit | 64 bit | 1977 年发布，已不安全（穷举可破解） |
| 3DES | 112/168 bit | 64 bit | 对 DES 加密三次，兼容性好但较慢 |
| AES | 128/192/256 bit | 128 bit | 当前国际标准，高效安全，广泛使用 |
| ChaCha20 | 256 bit | 流密码 | 流密码，适合移动端（无需硬件加速） |

AES (Advanced Encryption Standard) 是目前最重要的对称加密算法。它是一种**分组密码**，将明文分为固定大小的块（128 bit）逐块加密。AES 支持三种密钥长度：128/192/256 bit，分别需要 10/12/14 轮加密。每轮包含字节替换、行移位、列混合和密钥加四个步骤。

### 1.3 加密模式

分组密码一次只能加密一个固定大小的数据块。当明文超过一个块时，需要**加密模式**来定义如何将多个块串联起来。模式选择直接影响安全性和性能。

| 模式 | 全称 | 原理 | 安全性 | 特点 |
|------|------|------|--------|------|
| **ECB** | Electronic Codebook | 每个块独立加密 | 不安全 | 相同明文块→相同密文块，泄露数据模式 |
| **CBC** | Cipher Block Chaining | 每块先与前一块密文异或再加密 | 安全 | 需要初始化向量 IV，不可并行加密 |
| **CTR** | Counter Mode | 加密递增计数器，密文与明文异或 | 安全 | 可并行，流密码方式 |
| **GCM** | Galois/Counter Mode | CTR 模式 + GMAC 认证标签 | 安全 | 同时提供加密和完整性校验，HTTPS 首选 |

ECB 模式最简单但最不安全——经典的"ECB 企鹅"实验表明，用 ECB 加密位图图像后仍能看出原始图案。实际应用中应避免使用 ECB，优先选择 GCM（同时提供加密和认证）。

### 1.4 非对称加密算法

| 算法 | 数学基础 | 密钥长度 | 特点 |
|------|----------|----------|------|
| RSA | 大整数分解困难 | ≥ 2048 bit | 最广泛使用，密钥长、计算量大 |
| ECC | 椭圆曲线离散对数 | 256 bit ≈ RSA 3072 bit | 更短密钥达同等安全性，适合移动端和 IoT |
| Diffie-Hellman | 离散对数 | — | 密钥交换协议（不是加密算法），允许双方在不安全信道上协商共享密钥 |
| ElGamal | 离散对数 | — | 基于 DH，可用于加密和签名 |

**RSA 原理简述**：选择两个大素数 p、q，计算 n = p×q。公钥 (e, n)，私钥 (d, n)，其中 e×d ≡ 1 (mod φ(n))。加密：C = M^e mod n；解密：M = C^d mod n。安全性基于大整数分解的困难性——已知 n 要分解出 p 和 q 极其困难。

### 1.5 哈希函数

哈希函数（散列函数）将任意长度的输入映射为固定长度的输出（哈希值/摘要）。与加密不同，哈希是**单向**的——不存在解密过程，无法从哈希值恢复原文。

**三个核心性质**：
- **单向性（抗原像性）**：给定哈希值 h，计算上不可能找到输入 m 使得 H(m) = h
- **抗碰撞性**：计算上不可能找到两个不同的输入 m₁ ≠ m₂ 使得 H(m₁) = H(m₂)
- **雪崩效应**：输入即使只改变一个 bit，输出也会发生巨大变化（约 50% 的位翻转）

| 算法 | 输出长度 | 安全性 | 说明 |
|------|----------|--------|------|
| MD5 | 128 bit | 已被破解 | 2004 年王小云团队找到碰撞方法，不应再用于安全目的 |
| SHA-1 | 160 bit | 已被破解 | 2017 年 Google 公布实际碰撞，Git 等系统正在迁移 |
| SHA-256 | 256 bit | 安全 | SHA-2 家族，当前广泛使用（比特币、TLS 证书等） |
| SHA-3 | 可变 | 安全 | 2015 年发布，基于 Keccak 算法，与 SHA-2 结构完全不同 |

**常见应用**：
- **密码存储**：不存储明文密码，存储 hash(密码+盐值)。盐值（salt）是随机字符串，防止彩虹表攻击。推荐使用 bcrypt/scrypt/Argon2 等专用密码哈希函数（故意设计得很慢，增加暴力破解成本）
- **数据完整性校验**：下载文件后对比哈希值，确认文件未被篡改
- **数字签名**：对消息摘要签名而非对原文签名（效率高）
- **区块链**：每个区块包含前一区块的哈希，形成不可篡改的链

### 1.6 数字签名

数字签名是非对称加密的重要应用，用于证明消息的来源和完整性。它类似于手写签名的电子版，但更安全——无法伪造、不可否认。

**签名过程**：
1. 发送方对消息计算哈希摘要：digest = H(message)
2. 用发送方的**私钥**加密摘要：signature = E(digest, private_key)
3. 将消息 + 数字签名一起发送

**验证过程**：
4. 接收方用发送方的**公钥**解密签名：digest' = D(signature, public_key)
5. 接收方自己对消息计算哈希：digest = H(message)
6. 比对 digest' 和 digest，一致则验证通过

**数字签名保证三个性质**：
- **完整性**：消息未被篡改（因为篡改后哈希值会变）
- **认证**：消息确实来自声称的发送方（因为只有发送方有私钥）
- **不可否认性**：发送方不能抵赖（因为只有他能生成签名）

### 1.7 数字证书与 PKI

数字签名依赖公钥——但你怎么知道某个公钥确实属于你想通信的那个人？如果攻击者把自己的公钥冒充为银行的公钥给你，你用它加密发送的信息攻击者就能解密。这就是**公钥认证**问题。

**数字证书**：由受信任的第三方**CA (Certificate Authority，证书颁发机构)** 签发的电子文件，将公钥与其持有者身份绑定。证书包含：
- 持有者信息（域名、组织等）
- 持有者的公钥
- CA 用自己的私钥生成的数字签名
- 有效期
- 序列号、签名算法等元数据

当浏览器访问 HTTPS 网站时，服务器发送证书。浏览器用 CA 的公钥（预装在操作系统/浏览器中）验证 CA 的签名，从而确认服务器公钥的真实性。

**PKI (Public Key Infrastructure，公钥基础设施)**：管理数字证书的完整体系，包含：
- **CA (Certificate Authority)**：签发和管理证书
- **RA (Registration Authority)**：审核证书申请者的身份
- **证书仓库**：存储和分发证书
- **CRL (Certificate Revocation List) / OCSP**：证书吊销查询机制

**证书链 (Certificate Chain)**：为了安全，根 CA 的私钥极少直接签发终端证书。而是根 CA 签发中间 CA 的证书，中间 CA 再签发终端证书，形成链式结构：根 CA → 中间 CA → 终端证书。浏览器预装了根 CA 的证书（称为信任锚），沿证书链逐级验证。

## 二、网络安全

### 2.1 常见攻击与防御

网络攻击种类繁多，理解每种攻击的原理才能有效防御。以下是最常见的攻击类型：

| 攻击 | 原理 | 危害 | 防御 |
|------|------|------|------|
| **SQL 注入** | 在用户输入中插入恶意 SQL 语句，被服务器拼接执行 | 数据泄露、篡改、删库 | 参数化查询、ORM、输入验证、最小权限 |
| **XSS** (跨站脚本) | 在页面中注入恶意 JavaScript，在其他用户浏览器执行 | 窃取 Cookie、劫持会话、钓鱼 | 输出编码、CSP、HttpOnly Cookie |
| **CSRF** (跨站请求伪造) | 诱导已登录用户的浏览器发送未授权请求 | 以用户身份执行操作（如转账） | CSRF Token、SameSite Cookie、Referer 检查 |
| **DDoS** (分布式拒绝服务) | 利用大量僵尸主机同时发送请求耗尽服务器资源 | 服务不可用 | CDN、流量清洗、限流、弹性扩容 |
| **中间人攻击 (MITM)** | 攻击者在通信双方之间拦截和篡改数据 | 窃听、篡改通信内容 | HTTPS、证书锁定 (Certificate Pinning) |
| **DNS 劫持** | 篡改 DNS 解析结果，将域名指向恶意 IP | 钓鱼、流量劫持 | DNSSEC、DoH (DNS over HTTPS) |
| **ARP 欺骗** | 发送虚假 ARP 回复，将自己伪装为网关 | 局域网内窃听 | 静态 ARP 表、ARP 防护、802.1X |
| **重放攻击** | 截获并重发合法的认证数据 | 冒充合法用户 | 时间戳、Nonce、序列号 |

### 2.2 SQL 注入详解

SQL 注入是 Web 安全中最经典的攻击之一。当程序将用户输入直接拼接到 SQL 语句中执行时，攻击者可以通过精心构造的输入来改变 SQL 语句的语义。

**攻击示例**：

```sql
-- 假设后端代码：SELECT * FROM users WHERE name = '用户输入'
-- 攻击者输入: ' OR '1'='1
-- 实际执行：
SELECT * FROM users WHERE name = '' OR '1'='1'  -- 返回所有记录！

-- 更危险的输入: '; DROP TABLE users; --
-- 实际执行：
SELECT * FROM users WHERE name = ''; DROP TABLE users; --'  -- 删表！
```

**SQL 注入的类型**：
- **经典注入**：通过 UNION SELECT 直接获取数据
- **盲注 (Blind)**：页面不显示查询结果，通过布尔条件（页面是否正常）或时间延迟（SLEEP 函数）逐位推断数据
- **堆叠注入**：用分号分隔执行多条 SQL 语句
- **二次注入**：恶意数据先存入数据库，后续查询时触发

**防御措施**（按有效性排序）：
```python
# 1. 参数化查询（最有效，推荐）——将用户输入作为参数而非 SQL 的一部分
cursor.execute("SELECT * FROM users WHERE name = %s", (user_input,))

# 2. ORM 框架——自动参数化
User.objects.filter(name=user_input)

# 3. 输入验证——白名单校验（辅助手段，不可作为唯一防御）
# 4. 最小权限——数据库账号只给应用所需的最小权限（不给 DROP 权限）
# 5. WAF（Web 应用防火墙）——拦截常见攻击模式
```

### 2.3 XSS 攻击

XSS (Cross-Site Scripting) 攻击的核心是在受害者的浏览器中执行攻击者注入的 JavaScript 代码。由于脚本在受害网站的上下文中运行，它可以访问该网站的 Cookie、DOM 和其他敏感信息。

| 类型 | 攻击方式 | 持久性 | 示例 |
|------|----------|--------|------|
| **存储型 XSS** | 恶意脚本存入服务器数据库，每次页面加载都执行 | 持久 | 论坛帖子中注入 `<script>` 标签 |
| **反射型 XSS** | 恶意脚本通过 URL 参数传入，服务器将其反射到页面 | 非持久 | 构造恶意链接诱骗点击 |
| **DOM 型 XSS** | 前端 JavaScript 直接将不可信数据写入 DOM | 非持久 | `document.innerHTML = location.hash` |

**防御措施**：
- **输出编码**：将 `<` 转为 `&lt;`，`>` 转为 `&gt;` 等，防止浏览器将用户输入解析为 HTML/JS
- **CSP (Content Security Policy)**：通过 HTTP 头指定允许加载脚本的来源，禁止内联脚本
- **HttpOnly Cookie**：设置后 JavaScript 无法通过 `document.cookie` 读取，防止 Cookie 被窃取
- **输入验证**：在服务端对用户输入进行过滤和转义

### 2.4 CSRF 攻击

CSRF (Cross-Site Request Forgery) 利用的是**浏览器会自动携带 Cookie** 这一特性。攻击者构造一个恶意页面，当已登录目标网站的用户访问这个恶意页面时，浏览器会自动带上目标网站的 Cookie 发出请求，从而以用户身份执行操作。

**攻击场景**：用户已登录银行网站。攻击者发送一封包含以下 HTML 的邮件：
```html
<img src="https://bank.com/transfer?to=attacker&amount=10000" />
```
用户打开邮件时，浏览器自动请求这个 URL 并带上银行的 Cookie，转账就执行了。

**防御措施**：
- **CSRF Token**：服务器为每个表单生成随机 Token，提交时验证。攻击者无法获取这个 Token
- **SameSite Cookie**：设置 Cookie 的 SameSite 属性为 Strict 或 Lax，限制第三方网站携带 Cookie
- **Referer/Origin 检查**：验证请求来源是否为本站
- **双重提交 Cookie**：将 Token 同时放在 Cookie 和请求参数中

## 三、系统安全

### 3.1 缓冲区溢出

缓冲区溢出是最经典的系统级漏洞之一。当程序向固定大小的缓冲区写入超过其容量的数据时，多余的数据会覆盖缓冲区相邻的内存区域。在栈上，缓冲区之后通常是保存的寄存器值和**返回地址**。攻击者通过精心构造输入，覆盖返回地址为恶意代码的地址，从而在函数返回时劫持程序执行流。

**攻击过程**（栈溢出）：
1. 程序使用不安全的函数（如 C 语言的 `gets()`、`strcpy()`）读取用户输入
2. 用户输入超长数据，溢出缓冲区边界
3. 溢出的数据覆盖栈帧中的返回地址
4. 函数返回时跳转到攻击者指定的地址（如注入的 shellcode）

**防御措施**：

| 措施 | 层次 | 原理 |
|------|------|------|
| **栈保护 (Stack Canary)** | 编译器 | 在返回地址前放置一个随机值（金丝雀值），函数返回前检查该值是否被改变 |
| **ASLR** | 操作系统 | 地址空间布局随机化，每次运行时栈、堆、库的地址都不同，攻击者无法预知跳转目标 |
| **DEP/NX** | 硬件+OS | 数据执行保护/不可执行位，标记数据段为不可执行，即使注入了代码也无法运行 |
| **SafeSEH / CFI** | 编译器 | 控制流完整性保护，限制间接跳转的合法目标 |
| **使用安全函数** | 编程 | 用 `fgets()` 替代 `gets()`，用 `strncpy()` 替代 `strcpy()`，始终检查边界 |

**高级攻击技术**：
- **ROP (Return-Oriented Programming)**：绕过 DEP/NX，不注入新代码，而是复用程序中已有的代码片段（gadget）串联执行
- **堆溢出**：利用堆管理器的数据结构（如 free list）实现任意地址写

### 3.2 权限与访问控制

**最小权限原则 (Principle of Least Privilege)**：每个进程、用户、程序只应拥有完成其工作所必需的最小权限集合。例如，Web 服务器不应以 root 权限运行；数据库用户不应拥有 DROP TABLE 的权限。这样即使某个组件被攻陷，攻击者能造成的损害也被限制在最小范围。

**权限提升 (Privilege Escalation)**：
- **垂直提升**：低权限用户获得高权限（如普通用户 → root）。通常利用系统漏洞、SUID 程序漏洞或配置错误
- **水平提升**：用户 A 获得用户 B 的权限。通常利用应用层的访问控制缺陷（如通过修改 URL 中的用户 ID 访问他人数据）

## 四、访问控制模型

访问控制决定了"谁能对什么资源做什么操作"。不同的访问控制模型适用于不同的安全需求场景。

| 模型 | 全称 | 核心思想 | 典型应用 |
|------|------|----------|----------|
| **DAC** | Discretionary Access Control（自主访问控制） | 资源的所有者自行决定谁能访问。灵活但难以全局管控 | Linux 文件权限（rwx）、Windows NTFS |
| **MAC** | Mandatory Access Control（强制访问控制） | 由系统管理员统一制定安全策略，用户和程序不能改变。安全性高但灵活性低 | SELinux、军事/政府系统、Bell-LaPadula 模型 |
| **RBAC** | Role-Based Access Control（基于角色的访问控制） | 权限分配给角色，用户被分配到角色。简化了大规模权限管理 | 企业 ERP、云平台 IAM、数据库权限 |
| **ABAC** | Attribute-Based Access Control（基于属性的访问控制） | 基于用户属性、资源属性、环境属性的策略表达式来判断。最灵活 | AWS IAM Policy、零信任架构 |

**Bell-LaPadula 模型**（MAC 的经典模型）：
- **简单安全性质（不上读）**：低安全级别的主体不能读取高安全级别的客体
- **星性质（不下写）**：高安全级别的主体不能向低安全级别的客体写数据
- 目标：防止机密信息向下泄露

## 五、Web 安全

### 5.1 OWASP Top 10

OWASP (Open Web Application Security Project) 是一个开放的 Web 应用安全社区，其发布的 Top 10 是 Web 应用最关键的十大安全风险清单，是开发者和安全人员的重要参考：

1. **注入 (Injection)**：SQL/NoSQL/OS 命令/LDAP 注入，未对输入进行验证就拼接到命令中
2. **失效的身份认证**：弱密码策略、会话劫持、凭证填充攻击
3. **敏感数据暴露**：明文传输/存储密码、信用卡号等敏感信息
4. **XML 外部实体 (XXE)**：解析恶意 XML 时读取服务器文件或发起内网请求
5. **失效的访问控制**：越权访问——水平越权（访问他人数据）或垂直越权（执行管理员操作）
6. **安全配置错误**：使用默认密码、未禁用调试模式、暴露错误堆栈信息
7. **跨站脚本 (XSS)**：注入恶意脚本到页面中
8. **不安全的反序列化**：反序列化不可信数据导致远程代码执行
9. **使用含有已知漏洞的组件**：依赖的库/框架存在公开漏洞未修复
10. **不足的日志和监控**：攻击发生后无法及时发现和响应

### 5.2 HTTPS 与 TLS

HTTPS = HTTP + TLS (Transport Layer Security)。TLS 是 SSL 的继任者，为 HTTP 通信提供三层保护：

- **机密性**：使用对称加密（如 AES-256-GCM）加密所有传输数据
- **完整性**：使用 MAC (Message Authentication Code) 确保数据未被篡改
- **身份认证**：通过数字证书验证服务器身份，防止中间人攻击

**TLS 1.3 握手过程**（简化版）：
1. **Client Hello**：客户端发送支持的加密套件列表、随机数、密钥交换参数
2. **Server Hello**：服务器选择加密套件，发送证书、随机数、密钥交换参数
3. 客户端验证服务器证书（检查 CA 签名、域名、有效期）
4. 双方通过密钥交换算法（如 ECDHE）计算出共享的会话密钥
5. 后续通信使用会话密钥加密

TLS 1.3 相比 1.2 的改进：握手从 2-RTT 缩短到 1-RTT（甚至 0-RTT 恢复），移除了不安全的加密算法（如 RC4、SHA-1、RSA 密钥交换）。

### 5.3 同源策略与 CORS

**同源策略 (Same-Origin Policy)** 是浏览器最核心的安全机制。它规定：**协议 + 域名 + 端口** 三者完全相同才视为"同源"。

例如，`https://www.example.com:443` 与以下 URL 的同源关系：
- `https://www.example.com/path` → 同源（路径不影响）
- `http://www.example.com` → 不同源（协议不同）
- `https://api.example.com` → 不同源（域名不同）
- `https://www.example.com:8080` → 不同源（端口不同）

非同源的脚本**不能**：读取对方的 DOM、Cookie、LocalStorage，不能发送 AJAX 请求读取响应。

**CORS (Cross-Origin Resource Sharing)**：当前端需要跨域访问 API 时的标准解决方案。服务器通过设置 HTTP 响应头来声明允许哪些源访问：
- `Access-Control-Allow-Origin: https://frontend.example.com`
- `Access-Control-Allow-Methods: GET, POST, PUT`
- `Access-Control-Allow-Headers: Content-Type, Authorization`

**其他跨域方案**：
- JSONP：利用 `<script>` 标签不受同源策略限制的特性（只支持 GET，已不推荐）
- 代理服务器：前端请求同源的后端代理，由代理转发到目标服务器

### 5.4 认证与授权

**认证 (Authentication)** 和**授权 (Authorization)** 是两个不同的概念：
- 认证：验证"**你是谁**"——确认用户身份
- 授权：确定"**你能做什么**"——判断用户是否有权限执行某操作

**认证方式**：
- **密码认证**：最基础的方式。密码应使用专用哈希函数（bcrypt/scrypt/Argon2）加盐存储，绝不能存明文
- **多因子认证 (MFA)**：结合两种以上的认证因素——所知（密码）、所有（手机/硬件密钥）、所是（指纹/人脸）
- **OAuth 2.0**：第三方授权框架，允许用户授权第三方应用访问其在某服务上的资源，而无需共享密码。核心流程：用户在授权服务器登录 → 授权服务器返回授权码 → 第三方应用用授权码换取 Access Token → 用 Token 访问资源

**JWT (JSON Web Token)**：一种无状态的认证令牌格式，由三部分组成：
- **Header**：声明 Token 类型和签名算法（如 `{"alg": "HS256", "typ": "JWT"}`）
- **Payload**：包含声明（如用户 ID、过期时间等）
- **Signature**：对 Header + Payload 的签名，防止篡改

格式为 `Header.Payload.Signature`，每部分 Base64 编码。JWT 的优势是无状态——服务器不需要存储 Session，适合分布式系统和微服务架构。缺点是 Token 一旦签发无法主动失效（除非引入黑名单机制）。
