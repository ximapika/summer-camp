# 分布式系统

分布式系统不是传统本科核心课的固定考点，但在“计算机常识”、开放题、项目追问中很常见。重点掌握 CAP、一致性、复制、分片、缓存、消息队列、RPC、共识等概念。

---

## 一、基本概念

### 1.1 为什么需要分布式系统

- **扩展性**：单机资源有限，通过多机扩展吞吐。
- **高可用**：部分节点故障时系统仍可服务。
- **容错性**：通过冗余和恢复机制处理故障。
- **地理分布**：降低访问延迟，满足多地域需求。

### 1.2 分布式系统的难点

- 网络不可靠：延迟、丢包、乱序、分区。
- 时钟不可靠：不同机器时钟可能偏移。
- 节点会故障：宕机、重启、数据损坏。
- 并发复杂：多个副本同时读写，容易产生冲突。

## 二、CAP 与一致性

### 2.1 CAP 定理

在发生网络分区时，分布式系统不能同时完全满足：

- **Consistency 一致性**：所有节点读到同一份最新数据。
- **Availability 可用性**：每个请求都能得到非错误响应。
- **Partition tolerance 分区容忍性**：网络分区时系统仍能运行。

真实分布式系统必须考虑 P，因此常在 C 和 A 之间权衡。

### 2.2 一致性模型

| 模型 | 含义 |
|---|---|
| 强一致性 | 写入成功后，后续读取都能看到最新值 |
| 顺序一致性 | 所有操作表现得像按某个全局顺序执行 |
| 因果一致性 | 有因果关系的操作顺序被所有节点看到 |
| 最终一致性 | 如果没有新写入，副本最终会收敛 |
| 读己之写 | 用户能读到自己刚刚写入的数据 |

## 三、复制与分片

### 3.1 主从复制

- 主节点处理写请求，从节点复制主节点日志。
- 优点：实现简单，读扩展方便。
- 缺点：主节点故障需要选主；异步复制可能丢数据或读到旧值。

### 3.2 多主复制

- 多个节点都可写。
- 优点：可用性高、地域延迟低。
- 难点：写冲突检测与解决复杂。

### 3.3 分片 Sharding

把数据按规则拆到多个节点：

- **范围分片**：按 key 范围划分，范围查询友好，但可能热点。
- **哈希分片**：按 hash(key) 分布，负载均衡较好，但范围查询差。
- **一致性哈希**：节点增减时只迁移少量数据，常用于缓存集群。

## 四、共识算法

### 4.1 共识解决什么问题

共识算法让多个节点在故障条件下对某个值或日志顺序达成一致，常用于选主、元数据管理、复制日志。

### 4.2 Paxos 与 Raft

| 算法 | 特点 |
|---|---|
| Paxos | 理论经典，正确性强，但工程理解和实现较难 |
| Raft | 强调可理解性，分为 leader election、log replication、safety |

Raft 中的核心角色：

- **Leader**：处理客户端请求并复制日志。
- **Follower**：被动接收 leader 消息。
- **Candidate**：选举阶段的候选者。

### 4.3 脑裂

脑裂指多个节点都认为自己是主节点，可能导致数据冲突。常用多数派 quorum、租约、fencing token 等机制降低风险。

## 五、RPC 与服务通信

### 5.1 RPC

RPC（Remote Procedure Call）让调用远程服务看起来像调用本地函数。常见组成：

- 序列化/反序列化。
- 网络传输。
- 服务发现。
- 超时、重试、熔断、限流。

### 5.2 REST、gRPC 与消息队列

| 方式 | 特点 |
|---|---|
| REST | 基于 HTTP 和资源模型，简单通用 |
| gRPC | 基于 HTTP/2 和 Protobuf，高性能，适合服务间调用 |
| 消息队列 | 异步解耦、削峰填谷，但引入消息丢失/重复/顺序问题 |

## 六、缓存

### 6.1 缓存模式

- **Cache Aside**：应用先查缓存，未命中再查数据库并回填。
- **Write Through**：写缓存同时写数据库。
- **Write Back**：先写缓存，异步刷回数据库。

### 6.2 常见问题

- **缓存穿透**：查询不存在的数据，绕过缓存打到数据库。可用布隆过滤器、空值缓存。
- **缓存击穿**：热点 key 过期，大量请求同时访问数据库。可用互斥锁、逻辑过期。
- **缓存雪崩**：大量 key 同时过期或缓存集群故障。可用过期时间随机化、多级缓存。
- **缓存一致性**：数据库和缓存更新顺序不当会造成脏数据。

### 6.3 布隆过滤器

布隆过滤器用于判断元素是否可能存在：

- 判断“不存在”一定可靠。
- 判断“存在”可能误判。
- 优点是空间小，缺点是一般不支持直接删除。

## 七、消息队列

### 7.1 作用

- 异步处理。
- 服务解耦。
- 流量削峰。
- 事件驱动。

### 7.2 可靠性问题

- **至少一次**：消息不会丢，但可能重复。
- **至多一次**：消息不重复，但可能丢。
- **恰好一次**：语义最强，实现复杂，通常依赖幂等和事务机制。

消费者需要设计幂等逻辑，例如用业务唯一键去重。

## 八、分布式事务

### 8.1 2PC 两阶段提交

阶段：

1. Prepare：协调者询问参与者是否可以提交。
2. Commit/Rollback：协调者根据投票结果通知提交或回滚。

问题：同步阻塞、协调者单点、网络故障下处理复杂。

### 8.2 Saga

把长事务拆成多个本地事务，每步提供补偿操作。适合业务流程型事务，但只能保证最终一致性。

## 九、逻辑时钟与全局排序

### 9.1 物理时钟的问题

分布式系统中不同机器的物理时钟有偏差（clock skew），即使用 NTP 同步也只能达到毫秒级精度。因此不能依赖物理时钟判断事件先后顺序。

### 9.2 Lamport 逻辑时钟

规则：
1. 每个进程维护一个计数器 C，每次本地事件 C++
2. 发送消息时附带当前 C 值
3. 收到消息时 C = max(本地C, 消息C) + 1

**性质**：如果 a → b（a happens-before b），则 C(a) < C(b)。但反过来不成立——C(a) < C(b) 不能推出 a → b。

### 9.3 向量时钟 (Vector Clock)

每个进程维护一个长度为 n 的向量（n 为进程数）。

规则：
1. 本地事件：V[自己]++
2. 发送消息时附带整个向量
3. 收到消息：V[i] = max(V[i], 消息V[i])，然后 V[自己]++

**性质**：V(a) < V(b)（逐分量 ≤ 且至少一个 <）当且仅当 a → b。可以精确判断因果关系和并发。

### 9.4 Gossip 协议

节点像”八卦”一样传播信息：每个节点周期性地随机选择若干其他节点交换信息。

**特点**：
- 最终一致性：O(log n) 轮后所有节点收到信息
- 去中心化：无需 leader
- 容错性强：节点故障不影响传播
- 适用于：成员检测（如 SWIM）、状态传播、聚合计算

## 十、服务治理

### 10.1 服务发现

服务注册与发现：服务启动时注册到注册中心（如 Consul、Etcd、ZooKeeper），调用方从注册中心查找服务地址。

- **客户端发现**：客户端直接查询注册中心，自行负载均衡
- **服务端发现**：通过负载均衡器代理，客户端不感知服务实例

### 10.2 负载均衡

| 策略 | 特点 |
|------|------|
| 轮询 (Round Robin) | 简单均匀，不考虑机器性能差异 |
| 加权轮询 | 按权重分配，性能好的机器权重高 |
| 最少连接 | 发给当前连接数最少的机器 |
| 一致性哈希 | 按 key hash 路由，适合有状态服务或缓存 |
| 随机 | 简单，大量请求下近似均匀 |

### 10.3 熔断、限流、降级

**熔断 (Circuit Breaker)**：当下游服务错误率超过阈值时，暂时停止调用（打开熔断器），过一段时间半开尝试，恢复则关闭。防止故障扩散（雪崩效应）。状态：关闭 → 打开 → 半开 → 关闭。

**限流 (Rate Limiting)**：
- **令牌桶**：以固定速率生成令牌，请求消耗令牌，桶满则丢弃多余令牌。允许突发但有上限。
- **漏桶**：请求以固定速率流出，多余请求排队或丢弃。平滑流量。
- **滑动窗口计数器**：统计窗口内的请求数，超过阈值则拒绝。

**降级 (Degradation)**：系统压力过大时，暂时关闭非核心功能（如推荐、评论），保证核心功能可用。

### 10.4 分布式锁

常见实现：
- **Redis（SETNX + 过期时间）**：简单高效，但需注意过期时间设置和 Redis 主从切换导致的锁丢失
- **ZooKeeper（临时有序节点）**：可靠性更高，客户端断连自动释放
- **Etcd（基于 Raft 共识）**：强一致性

**注意事项**：
- 必须设置过期时间（防止持有者宕机后锁永不释放）
- 过期时间不能太短（业务未完成锁就过期）
- Redlock 方案在多 Redis 实例上加锁，增加可靠性
- 锁续期：Watchdog 机制自动续期（如 Redisson）

## 十一、容器与编排基础

### 11.1 容器 (Container)

容器是基于 Linux namespace 和 cgroups 的轻量级隔离环境：
- **namespace**：隔离进程ID、网络、文件系统、用户等
- **cgroups**：限制 CPU、内存、I/O 等资源使用

容器 vs 虚拟机：容器共享宿主机内核，启动快（秒级）、开销小；虚拟机有独立内核，隔离性更强但更重。

**Docker**：最常用的容器运行时。Dockerfile 定义镜像构建过程，镜像是不可变的分层文件系统。

### 11.2 Kubernetes (K8s) 核心概念

| 概念 | 说明 |
|------|------|
| **Pod** | K8s 最小调度单位，包含一个或多个容器，共享网络和存储 |
| **Service** | 为一组 Pod 提供稳定的访问入口（虚拟 IP + 负载均衡） |
| **Deployment** | 管理 Pod 的副本数、滚动更新、回滚 |
| **ConfigMap / Secret** | 配置和敏感信息管理 |
| **Ingress** | HTTP/HTTPS 路由入口 |
| **HPA** | 根据 CPU/内存等指标自动伸缩 Pod 数量 |
