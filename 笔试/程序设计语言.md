# 程序设计语言

程序设计语言常识常出现在笔试的概念辨析题或面试追问中，重点不是某一门语言的语法细节，而是类型、作用域、内存、面向对象、函数式、并发模型等通用概念。

---

## 一、语言分类与范式

### 1.1 编译型、解释型与 JIT

| 类型 | 特点 | 典型语言/环境 |
|---|---|---|
| 编译型 | 源代码先编译成机器码或目标文件，运行时开销较低 | C/C++、Rust、Go |
| 解释型 | 解释器逐步执行源代码或字节码，开发灵活 | Python、JavaScript |
| 字节码 + 虚拟机 | 源代码先编译成平台无关字节码，由 VM 执行 | Java、Kotlin、C# |
| JIT | 运行时把热点代码动态编译成本地机器码 | JVM、V8、PyPy |

注意：一门语言不一定天然“编译”或“解释”，实际取决于实现。例如 Python 也会生成 `.pyc` 字节码，JavaScript 在 V8 中大量使用 JIT。

### 1.2 常见编程范式

- **过程式**：以过程/函数和顺序控制为中心，如 C。
- **面向对象**：以对象、封装、继承、多态为中心，如 Java、C++。
- **函数式**：强调不可变数据、纯函数、高阶函数，如 Haskell、部分 Scala。
- **声明式**：描述“要什么”而不是“怎么做”，如 SQL、正则表达式。
- **泛型编程**：用类型参数抽象算法和数据结构，如 C++ template、Java generics。

## 二、类型系统

### 2.1 静态类型 vs 动态类型

| 对比 | 静态类型 | 动态类型 |
|---|---|---|
| 类型检查时间 | 编译期 | 运行期 |
| 优点 | 更早发现错误，利于优化和 IDE 支持 | 灵活，原型开发快 |
| 缺点 | 类型声明和约束更多 | 运行期类型错误风险更高 |
| 例子 | C/C++、Java、Go、Rust | Python、JavaScript、Ruby |

### 2.2 强类型 vs 弱类型

- **强类型**：不同类型之间不会轻易隐式转换，如 Python 中 `"1" + 1` 报错。
- **弱类型**：允许较多隐式转换，如 JavaScript 中 `"1" + 1` 得到 `"11"`。

静态/动态关注“何时检查”，强/弱关注“是否允许隐式混用”，二者不是一回事。

### 2.3 类型推断与泛型

- **类型推断**：编译器根据上下文推导类型，如 C++ `auto`、Rust `let x = 1`。
- **泛型**：让代码对多种类型复用，同时保留类型安全。
- **协变/逆变**：描述泛型类型在子类型关系下如何变化，常见于 Java/C# 的集合与函数类型。

## 三、作用域、生命周期与绑定

### 3.1 作用域

- **词法作用域**：变量可见性由代码书写位置决定，现代语言普遍采用。
- **动态作用域**：变量解析依赖调用链，较少见。

### 3.2 生命周期

- **自动存储期**：局部变量随栈帧创建和销毁。
- **静态存储期**：全局变量、静态变量在程序生命周期内存在。
- **动态存储期**：堆对象由 `new/malloc` 或运行时分配，释放方式取决于语言。

### 3.3 值传递、引用传递与移动语义

- **值传递**：传入副本，修改副本不影响原对象。
- **引用/指针传递**：传入对象别名或地址，可修改原对象。
- **移动语义**：转移资源所有权，避免昂贵拷贝，C++11 后很重要。

## 四、内存管理

### 4.1 栈与堆

| 区域 | 特点 |
|---|---|
| 栈 | 分配/释放快，随函数调用自动管理，空间有限 |
| 堆 | 生命周期灵活，需手动释放或由 GC 管理，碎片和泄漏风险更高 |

### 4.2 手动内存管理

C/C++ 中常见问题：

- **内存泄漏**：分配后未释放。
- **悬空指针**：对象释放后仍被访问。
- **重复释放**：同一块内存被释放多次。
- **越界访问**：数组或缓冲区访问超出范围。

C++ 常用 RAII 和智能指针降低风险：

- `unique_ptr`：独占所有权。
- `shared_ptr`：引用计数共享所有权。
- `weak_ptr`：弱引用，避免循环引用。

### 4.3 垃圾回收 GC

常见 GC 思路：

- **引用计数**：对象引用数为 0 时释放；问题是循环引用。
- **标记-清除**：从根对象出发标记可达对象，清除不可达对象。
- **复制算法**：把存活对象复制到新区域，回收旧区域。
- **分代 GC**：基于“大部分对象朝生夕死”的假设，将对象分为新生代/老年代。

## 五、面向对象

### 5.1 三大特性

- **封装**：隐藏内部实现，通过接口访问状态。
- **继承**：复用和扩展已有类型。
- **多态**：同一接口在不同对象上表现不同。

### 5.2 静态多态 vs 动态多态

- **静态多态**：编译期决定，如函数重载、模板。
- **动态多态**：运行期决定，如虚函数、接口派发。

### 5.3 重载、重写与隐藏

- **重载 overload**：同一作用域内函数名相同、参数列表不同。
- **重写 override**：子类重新实现父类虚函数。
- **隐藏 hide**：子类同名成员遮蔽父类成员，不一定构成重写。

## 六、函数式编程

### 6.1 核心概念

- **纯函数**：输出只依赖输入，无副作用。
- **不可变性**：数据创建后不修改，降低并发和推理成本。
- **高阶函数**：函数可作为参数或返回值。
- **闭包**：函数捕获其定义环境中的变量。
- **柯里化**：把多参数函数转换为一系列单参数函数。

### 6.2 Lambda 与闭包

Lambda 是匿名函数；闭包是“函数 + 捕获环境”。C++ lambda 的 `[&]`、`[=]` 分别表示按引用/按值捕获。

## 七、错误处理

| 方式 | 特点 | 例子 |
|---|---|---|
| 返回错误码 | 简单直接，但容易忘记检查 | C |
| 异常 | 正常逻辑和错误逻辑分离，但控制流不显式 | Java、C++、Python |
| Option/Result | 把错误纳入类型系统，调用者必须处理 | Rust、Scala |

常见概念：异常安全、资源释放、`finally`/defer、RAII。

## 八、并发模型

### 8.1 线程与锁

- 多线程共享地址空间，通信方便，但需要锁保护共享数据。
- 常见问题：竞态条件、死锁、活锁、优先级反转。

### 8.2 Actor、协程与 async/await

- **Actor**：每个 Actor 拥有私有状态，通过消息通信。
- **协程**：用户态轻量级执行单元，适合高并发 I/O。
- **async/await**：用同步风格书写异步逻辑，本质通常是状态机。

### 8.3 内存模型

内存模型规定多线程下读写的可见性和重排序规则。需要理解：

- 原子操作 atomic。
- happens-before。
- acquire/release。
- volatile 不等于线程安全。

## 九、重要语言特性专题

### 9.1 C++ 虚函数与多态实现

C++ 动态多态通过**虚函数表 (vtable)** 实现：

- 每个含虚函数的类有一张虚函数表，存放虚函数的地址
- 每个对象有一个隐藏的 **vptr** 指向所属类的 vtable
- 调用虚函数时：通过 vptr 查 vtable → 找到函数地址 → 间接调用

**性能代价**：一次额外的指针解引用（通常可忽略），且虚函数无法内联。

**纯虚函数**：`virtual void f() = 0;`，含纯虚函数的类为抽象类，不能实例化。

**析构函数为什么要声明为 virtual？** 通过基类指针删除派生类对象时，如果析构函数不是虚的，只会调用基类析构函数，派生类资源不会释放（内存泄漏）。

### 9.2 Rust 所有权与生命周期

Rust 通过所有权系统在**编译期**保证内存安全，无需 GC：

**三条规则**：
1. 每个值有且仅有一个所有者（owner）
2. 值在所有者离开作用域时自动释放（drop）
3. 所有权可以转移（move），转移后原变量不可用

**借用 (Borrowing)**：
- `&T`：不可变借用，可以有多个
- `&mut T`：可变借用，同一时刻只能有一个
- 不可变借用和可变借用不能同时存在

**生命周期 (Lifetime)**：编译器通过生命周期标注 `'a` 确保引用不会超过被引用对象的存活时间（防止悬空引用）。

**对比 C++**：C++ 有 RAII 和智能指针，但仍然可以出现悬空引用、数据竞争。Rust 在编译期就杜绝这些问题。

### 9.3 Go 并发模型

Go 的并发基于 **CSP (Communicating Sequential Processes)** 模型：

- **goroutine**：轻量级协程（初始栈约 2-8 KB，可动态增长），由 Go 运行时调度
- **channel**：goroutine 之间通信的管道，类型安全
- 设计哲学：**"不要通过共享内存来通信，而是通过通信来共享内存"**

**GMP 调度模型**：
- **G (Goroutine)**：用户级协程
- **M (Machine)**：操作系统线程
- **P (Processor)**：逻辑处理器，维护本地 goroutine 队列

Go 运行时将大量 G 映射到少量 M 上（M:N 调度），P 的数量默认等于 CPU 核心数。

### 9.4 Python GIL

**GIL (Global Interpreter Lock)**：CPython 解释器的全局互斥锁，同一时刻只有一个线程执行 Python 字节码。

**影响**：
- CPU 密集型任务：多线程无法利用多核，应使用 `multiprocessing`（多进程）
- I/O 密集型任务：线程在等待 I/O 时会释放 GIL，多线程仍有效
- `asyncio`：单线程异步 I/O，适合大量网络请求

**为什么有 GIL？** 简化 CPython 的内存管理（引用计数不需要加锁）。其他 Python 实现（如 Jython、PyPy STM）没有 GIL。Python 3.13 开始实验性支持 no-GIL 模式。

### 9.5 Java JVM 核心概念

**JVM 内存区域**：
- **堆 (Heap)**：对象实例存放区域，GC 的主要管理区域
- **方法区 / 元空间**：类信息、常量池、静态变量
- **虚拟机栈**：每个线程有自己的栈，存放栈帧（局部变量表、操作数栈、返回地址）
- **程序计数器**：当前线程执行的字节码行号
- **本地方法栈**：JNI 调用的原生方法

**类加载机制**：
- 加载 → 验证 → 准备 → 解析 → 初始化
- **双亲委派模型**：类加载请求先委派给父加载器，父加载器无法完成时才自己加载。保证核心类（如 java.lang.Object）只被加载一次。

**JIT 编译**：JVM 在运行时识别热点代码（方法调用计数或循环回边计数超过阈值），将字节码编译为本地机器码。C1 编译器（快速编译）和 C2 编译器（深度优化）。

### 9.6 语言对比总结

| 特性 | C/C++ | Java | Python | Go | Rust |
|------|-------|------|--------|-----|------|
| 类型系统 | 静态/弱 | 静态/强 | 动态/强 | 静态/强 | 静态/强 |
| 内存管理 | 手动/RAII | GC | GC (引用计数+分代) | GC (三色标记) | 所有权系统 |
| 并发模型 | 线程+锁 | 线程+锁+并发库 | GIL/多进程/asyncio | goroutine+channel | async/await+线程 |
| 运行方式 | 编译 | 字节码+JIT | 解释+JIT(PyPy) | 编译 | 编译 |
| 空指针 | 有（危险） | 有（NPE） | 无（None） | 有（nil） | 无（Option） |
| 异常处理 | 异常 | 异常 | 异常 | 返回 error | Result/panic |

## 十、常见高频问题

1. 静态类型和动态类型的区别是什么？
2. 编译型语言一定比解释型语言快吗？（不一定，JIT 可以做运行时优化）
3. C++ 中 `new/delete` 和 `malloc/free` 的区别是什么？（前者调用构造/析构函数）
4. `unique_ptr`、`shared_ptr`、`weak_ptr` 分别解决什么问题？
5. 什么是闭包？闭包捕获变量有什么风险？
6. 重载、重写、隐藏如何区分？
7. GC 的引用计数为什么处理不了循环引用？
8. 协程和线程有什么区别？
9. `volatile` 能保证多线程安全吗？（不能，volatile 只保证可见性，不保证原子性）
10. Java/C++/Python 在内存管理上的主要差异是什么？
11. C++ 虚函数表的机制是什么？析构函数为什么要 virtual？
12. Rust 的所有权系统如何保证内存安全？
13. Go 的 goroutine 和 Java 线程有什么区别？
14. Python 的 GIL 是什么？如何绕过？
15. Java 的双亲委派模型是什么？
