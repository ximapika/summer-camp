# 操作系统

## 一、进程管理

### 1.1 进程与线程

**进程 (Process)** 是程序的一次执行实例。当你双击打开一个程序时，操作系统就为它创建了一个进程。进程是操作系统进行**资源分配**的基本单位——每个进程拥有独立的内存空间、文件描述符等资源，互不干扰。

**线程 (Thread)** 是进程内部的一条执行流。一个进程可以包含多个线程，它们共享进程的内存和资源，但各自有独立的栈和程序计数器。线程是 CPU **调度**的基本单位——操作系统实际上是在调度线程，而非进程。

| 对比项 | 进程 (Process) | 线程 (Thread) |
|--------|---------------|---------------|
| 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 地址空间 | 独立的地址空间 | 共享所属进程的地址空间 |
| 资源 | 拥有独立资源（内存、文件描述符等） | 共享进程资源，拥有独立的栈和寄存器 |
| 创建/切换开销 | 大（需要分配/切换地址空间） | 小（只需切换栈和寄存器） |
| 通信 | 需要 IPC 机制（管道、共享内存等） | 可直接读写共享变量（需同步） |
| 安全性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |

**协程 (Coroutine)**：用户态的轻量级线程。与线程的区别在于，协程的调度由程序自身控制（协作式），而非由操作系统内核调度（抢占式）。协程切换不需要系统调用，开销极小（仅保存/恢复少量寄存器），适合 I/O 密集型场景。典型实现：Go 的 goroutine、Python 的 asyncio、Kotlin 的 coroutine。

**进程控制块 (PCB, Process Control Block)**：操作系统为每个进程维护的数据结构，是进程存在的唯一标识。PCB 中包含了操作系统管理进程所需的全部信息：

- **进程标识符 (PID)**：系统中唯一标识进程的整数
- **进程状态**：就绪、运行、阻塞等
- **程序计数器 (PC)**：下一条要执行的指令地址
- **CPU 寄存器集合**：进程切换时需要保存/恢复的寄存器值
- **调度信息**：优先级、调度队列指针、已使用的 CPU 时间
- **内存管理信息**：页表基址寄存器、段表、地址空间大小
- **I/O 状态信息**：打开的文件列表、分配的 I/O 设备
- **记账信息**：CPU 使用时间、时间限制、进程号

### 1.2 进程状态转换

进程在生命周期中会在不同状态之间转换。最基本的是三状态模型：

```
         创建
          │
          ▼
       就绪(Ready) ◄──────── 时间片用完/被抢占
          │                        ▲
     被调度 │                        │
          ▼                        │
       运行(Running) ──────────────┘
          │
    等待事件 │
          ▼
       阻塞(Blocked) ──── 事件完成 ──→ 就绪(Ready)
```

- **就绪 (Ready)**：进程已获得除 CPU 以外的所有资源，等待被调度执行
- **运行 (Running)**：进程正在 CPU 上执行指令
- **阻塞 (Blocked/Waiting)**：进程因等待某个事件（如 I/O 完成、信号量）而暂停执行

五状态模型增加了：
- **新建 (New)**：进程刚被创建，PCB 已建立但尚未进入就绪队列
- **终止 (Terminated)**：进程执行完毕或被杀死，等待操作系统回收资源

**注意**：阻塞→运行 的转换是不存在的。阻塞的进程事件完成后先回到就绪态，再等待调度。

### 1.3 进程调度算法

进程调度决定了哪个就绪进程获得 CPU。调度算法直接影响系统的吞吐量、响应时间和公平性。

| 算法 | 全称 | 核心思想 | 是否抢占 | 优缺点 |
|------|------|----------|----------|--------|
| FCFS | First Come First Served | 按到达顺序执行，先来先服务 | 非抢占 | 简单公平，但短作业可能等很久（护航效应） |
| SJF | Shortest Job First | 优先执行预计运行时间最短的作业 | 非抢占 | 平均等待时间最小，但难以预知运行时间，可能饥饿 |
| SRTF | Shortest Remaining Time First | SJF 的抢占版本，新到达的短作业可抢占当前执行的长作业 | 抢占 | 理论最优，但开销大 |
| RR | Round Robin | 每个进程执行一个时间片(quantum)后轮换，保证每个进程都能获得 CPU 时间 | 抢占 | 公平，时间片太小→切换开销大，太大→退化为FCFS |
| 优先级调度 | Priority Scheduling | 按优先级高低调度，优先级高的先执行 | 均可 | 灵活，但低优先级进程可能饥饿（解决：老化机制） |
| MLFQ | Multi-Level Feedback Queue | 多个优先级队列，根据进程行为动态调整优先级 | 抢占 | 综合性能最好，兼顾交互性和吞吐量 |

**MLFQ 核心思想**：这是现代操作系统（如 Linux CFS 的前身）最常用的调度框架：
- 设置多个优先级队列，优先级越高的队列时间片越短
- 新进程进入最高优先级队列（假设它是短进程/交互式进程）
- 如果进程在一个时间片内完成，说明它确实是短进程，获得了好的响应时间
- 如果进程用完时间片还没完成，降入下一级队列（说明它可能是长进程）
- 在低优先级队列等待过久的进程可提升优先级（防止饥饿）
- 效果：交互式短进程获得快速响应，长进程也不会被完全饿死

### 1.4 进程同步与互斥

当多个进程/线程需要访问共享资源时，如果不加控制，可能会产生**竞态条件 (Race Condition)**——最终结果取决于进程执行的时序，导致不确定的错误。

**临界区 (Critical Section)**：访问共享资源的代码段。任何时刻最多只能有一个进程在临界区内执行。

**互斥 (Mutual Exclusion)**：保证同一时刻只有一个进程访问临界区。
**同步 (Synchronization)**：保证多个进程按照特定的顺序执行。

**临界区问题的解决要求**：
1. **互斥**：同一时刻只有一个进程在临界区
2. **前进 (Progress)**：临界区空闲时，不能无限期阻止进程进入（不能死等）
3. **有限等待 (Bounded Waiting)**：进程请求进入临界区后，必须在有限时间内获准进入（不能饿死）

#### 锁的实现

**自旋锁 (Spinlock)**：最简单的锁实现，等待锁时 CPU 忙等（自旋）。实现依赖硬件提供的**原子指令**：

**Test-and-Set (TAS)**：原子地读取旧值并设为 1。
```
// 硬件原子操作
int test_and_set(int *lock) {
    int old = *lock;
    *lock = 1;
    return old;  // 返回旧值
}

// 加锁：如果旧值为0（未锁），则加锁成功
void acquire(int *lock) {
    while (test_and_set(lock) == 1)
        ;  // 自旋等待
}
void release(int *lock) { *lock = 0; }
```

**Compare-and-Swap (CAS)**：比较旧值，相等则替换为新值，返回旧值。是实现无锁数据结构的基础。
```
int compare_and_swap(int *addr, int expected, int new_val) {
    int old = *addr;
    if (old == expected) *addr = new_val;
    return old;
}
```

**自旋锁的问题**：等待时 CPU 空转浪费资源。适合临界区很短的场景（如内核中保护共享数据结构）。对于长临界区，应使用让权等待（yield / sleep）的锁。

#### 条件变量 (Condition Variable)

条件变量用于线程之间的**等待-通知**机制，必须与锁配合使用。

- **wait(cv, lock)**：原子地释放锁并进入睡眠，被唤醒后重新获取锁
- **signal(cv)**：唤醒一个在该条件变量上等待的线程
- **broadcast(cv)**：唤醒所有等待的线程

**使用模式**（Mesa 语义，需要 while 循环检查条件）：
```
lock.acquire();
while (!condition)        // 必须用 while，不能用 if
    cv.wait(&lock);       // 释放锁并睡眠，唤醒后重新获锁
// ... 执行操作 ...
cv.signal();              // 唤醒等待者
lock.release();
```

**为什么用 while 不用 if**：Mesa 语义下，signal 只是"提示"而非"保证"——被唤醒的线程重新获取锁后，条件可能已被其他线程改变（虚假唤醒）。

**信号量 (Semaphore)**：由 Dijkstra 提出的经典同步工具。信号量 S 是一个整数变量，只能通过 P 和 V 两个原子操作访问：

- **P 操作 (wait/down)**：S--；若 S < 0，当前进程阻塞（入等待队列）
- **V 操作 (signal/up)**：S++；若 S ≤ 0，唤醒等待队列中的一个进程

信号量的值的含义：
- S > 0：还有 S 个可用资源
- S = 0：无可用资源，也无等待进程
- S < 0：|S| 个进程正在等待

用法：
- **互斥信号量**：初值为 1，P(mutex)...临界区...V(mutex)
- **同步信号量**：初值为 0，进程 A 先 V(S)，进程 B 后 P(S)，保证 A 在 B 之前执行

**经典同步问题**：

1. **生产者-消费者问题**：生产者向缓冲区放产品，消费者从缓冲区取产品。需要三个信号量：
   - mutex = 1（互斥访问缓冲区）
   - empty = n（空缓冲区数量，初始为 n）
   - full = 0（满缓冲区数量，初始为 0）
   - 生产者：P(empty) → P(mutex) → 放产品 → V(mutex) → V(full)
   - 消费者：P(full) → P(mutex) → 取产品 → V(mutex) → V(empty)
   - **注意**：P(empty/full) 必须在 P(mutex) 之前，否则可能死锁

2. **读者-写者问题**：多个读者可以同时读，但写者必须独占访问。
   - 读者优先：第一个读者加写锁，最后一个读者解写锁。用 readcount 计数器 + mutex 保护计数器。
   - 写者优先：需要额外信号量，新读者在有写者等待时不能进入。

3. **哲学家就餐问题**：5 个哲学家围桌，5 根筷子，每人需要左右两根筷子才能吃饭。
   - 直接取会导致死锁（每人都拿了左筷等右筷）
   - 解决方案：限制最多 4 人同时取筷 / 奇数先左后右、偶数先右后左 / 一次性拿起两根

**管程 (Monitor)**：一种高级同步机制，将共享变量及其操作封装在一个模块中。管程内部自动保证互斥（同一时刻只有一个进程在管程内执行），通过条件变量的 wait() 和 signal() 实现进程同步。Java 的 synchronized 关键字就是管程的一种实现。

#### 生产者-消费者问题的条件变量实现

```
mutex lock;
condition_variable not_full, not_empty;
BoundedBuffer buffer;  // 有界循环缓冲区

Producer:                        Consumer:
  lock.acquire();                  lock.acquire();
  while (buffer.isFull())          while (buffer.isEmpty())
      not_full.wait(&lock);            not_empty.wait(&lock);
  buffer.enqueue(item);            item = buffer.dequeue();
  not_empty.signal();              not_full.signal();
  lock.release();                  lock.release();
```

**与信号量实现的对比**：条件变量需要显式的锁 + while 检查，语义更清晰；信号量将同步和互斥统一在一个机制中，更简洁但更易出错（P 操作顺序错误会死锁）。

### 1.5 死锁

**死锁 (Deadlock)** 是指两个或多个进程相互等待对方持有的资源，导致所有进程都无法继续执行。例如：进程 A 持有资源 1 等待资源 2，进程 B 持有资源 2 等待资源 1。

**四个必要条件**（同时满足才会死锁）：
1. **互斥条件**：资源一次只能被一个进程使用（如打印机）
2. **持有并等待 (Hold and Wait)**：进程持有至少一个资源，同时等待获取其他资源
3. **不可抢占 (No Preemption)**：已分配给进程的资源不能被强行剥夺，只能由进程自愿释放
4. **循环等待 (Circular Wait)**：存在进程资源的循环等待链 P1→P2→...→Pn→P1

**处理策略**：

| 策略 | 方法 | 特点 |
|------|------|------|
| **预防** | 破坏四个必要条件之一。如：一次性申请所有资源（破坏持有并等待）、按编号顺序申请（破坏循环等待） | 限制过强，资源利用率低 |
| **避免** | 在资源分配前判断是否安全（银行家算法） | 需要预知最大需求，开销大 |
| **检测+恢复** | 允许死锁发生，定期检测（资源分配图检测环），发现后终止进程或回滚 | 实现复杂 |
| **忽略** | 鸵鸟策略——假装死锁不会发生 | 大多数实际 OS 采用（死锁概率低，处理代价高） |

**银行家算法**（死锁避免）：
- 类比银行贷款：银行家（OS）不会把所有现金（资源）都借出去，总要留够能让至少一个客户（进程）完成的量
- 数据结构：
  - Available[m]：每类资源当前可用数量
  - Max[n][m]：每个进程对每类资源的最大需求
  - Allocation[n][m]：每个进程当前已分配的资源
  - Need[n][m] = Max - Allocation：每个进程还需要的资源
- **安全性检查**：尝试找到一个**安全序列**——按这个顺序依次满足每个进程的最大需求，每个进程完成后释放资源供后续进程使用。如果存在安全序列，系统处于安全状态，可以分配资源。
- 安全状态一定不会死锁，不安全状态不一定死锁但有风险。

## 二、内存管理

### 2.1 内存分配方式

操作系统需要将程序加载到内存中才能执行。内存管理的核心目标是实现**地址空间隔离**——每个进程认为自己独占内存，互不干扰。

#### 地址转换的基本机制

**Base-and-Bound（基址-界限）**：最简单的地址转换方案。
- 每个进程分配一段连续物理内存，硬件用两个寄存器控制：
  - **Base 寄存器**：起始物理地址
  - **Bound 寄存器**：地址空间长度（或上界地址）
- 转换：物理地址 = 虚拟地址 + Base（前提：虚拟地址 < Bound，否则触发保护异常）
- 缺点：进程必须占据连续物理内存，且整个地址空间必须驻留在内存中

**分段（Segmentation）**：将地址空间按逻辑结构分为多个段（代码段、数据段、栈段、堆段等），每个段独立映射。
- **段表 (Segment Map)**：每项包含 Base（段基址）、Limit（段长度）、Valid（是否有效）等字段
- 转换：物理地址 = Segment[seg_num].base + offset（需检查 offset < limit 且 valid = 1）
- 优点：按逻辑结构划分，便于共享和保护
- 缺点：段长度可变，产生外部碎片

**连续分配**：每个进程占据一段连续的内存区域。

| 方式 | 描述 | 碎片 |
|------|------|------|
| 单一连续 | 整个用户区分给一个进程，最简单 | 内部碎片（分配的空间大于实际需要） |
| 固定分区 | 内存预先分为若干固定大小的区域 | 内部碎片（进程小于分区大小的部分浪费） |
| 动态分区 | 按进程实际大小动态分配连续空间 | 外部碎片（空闲区域被分割成小块，总量够但不连续） |

**内部碎片 vs 外部碎片**：
- 内部碎片：分配给进程的内存中，进程没有使用的部分（分多了）
- 外部碎片：内存中未分配的小空间，太小无法满足任何请求（碎了）
- 解决外部碎片：紧凑（移动进程合并空闲区，代价大）或分页

**动态分区分配算法**：
- **首次适应 (First Fit)**：从低地址开始，找第一个足够大的空闲区。综合效果最好，倾向于保留高地址的大空闲区。
- **最佳适应 (Best Fit)**：找最小的够大的空闲区。节省大空闲区，但产生大量难以利用的小碎片。
- **最坏适应 (Worst Fit)**：找最大的空闲区分配。避免小碎片，但大空闲区会被快速用完。
- **邻近适应 (Next Fit)**：从上次分配位置继续查找。比首次适应更快，但可能导致高地址碎片化。

### 2.2 分页存储 (Paging)

分页是现代操作系统最重要的内存管理技术。核心思想是将内存空间"打碎"为固定大小的块来管理，消除外部碎片。

- 将进程的逻辑地址空间划分为固定大小的**页 (Page)**（通常 4KB）
- 将物理内存划分为同样大小的**页框/帧 (Frame)**
- 页和页框大小相同，但页不必占据连续的页框——通过页表映射实现离散存放

**逻辑地址结构**：| 页号 P | 页内偏移 W |
- 页号 = 逻辑地址 / 页大小（整除）
- 页内偏移 = 逻辑地址 % 页大小（取余）

**页表 (Page Table)**：每个进程一张，记录页号到页框号的映射关系。地址转换过程：
1. CPU 产生逻辑地址
2. 取出页号 P，查页表得到对应的页框号 F
3. 物理地址 = F × 页大小 + W（页内偏移不变）

**页表项 (PTE, Page Table Entry) 的标志位**：

每个 PTE 不仅存储页框号，还包含多个控制位，是虚拟内存实现各种高级功能的基础：

| 标志位 | 含义 | 用途 |
|--------|------|------|
| **Valid (V)** | 该页表项是否有效 | V=0 时访问触发 Page Fault，用于实现 Demand Paging |
| **Read (R)** | 是否允许读 | 权限控制 |
| **Write (W)** | 是否允许写 | 设为只读可实现 Copy-on-Write |
| **Dirty (D)** | 该页是否被修改过 | 页面置换时决定是否需要写回磁盘 |
| **Accessed/Reference (A)** | 该页是否被访问过 | Clock 算法等置换算法使用 |
| **User/Supervisor (U/S)** | 用户态是否可访问 | 内核页面保护 |
| **Execute (X/NX)** | 是否允许执行 | 防止数据页被当作代码执行（安全机制） |

**多级页表**：当地址空间很大时（如 64 位系统），一张页表本身就非常大。解决方案是将页表本身也分页——形成多级页表结构。

**32 位系统二级页表（10-10-12）**：
- 逻辑地址 = | 10 位一级页号 | 10 位二级页号 | 12 位偏移 |
- 一级页表（页目录，1024 项）指向二级页表
- 每个二级页表（1024 项）指向实际页框
- 每个页表刚好占一个 4KB 页，方便管理
- 优点：只有实际使用的虚拟地址范围才需要分配二级页表，大量未使用的地址空间不占内存

**x86_64 四级页表（9-9-9-9-12）**：
- 虚拟地址 48 位有效（高 16 位为符号扩展），物理地址最多 52 位
- 地址结构：| 9 位 PML4 | 9 位 PDPT | 9 位 PD | 9 位 PT | 12 位偏移 |
- 四级结构：PML4（Page Map Level 4）→ PDPT（Page Directory Pointer Table）→ PD（Page Directory）→ PT（Page Table）→ 物理页框
- 每级 512 项（2⁹），每项 8 字节，每个表 4KB = 1 页
- **CR3 寄存器**存储当前进程的 PML4 表物理地址，进程切换时切换 CR3 即切换整个地址空间

**多级页表实现内存共享**：不同进程的页表可以在某一级指向相同的下级页表或相同的物理页框，从而实现代码段共享、共享内存等功能。

**TLB (Translation Lookaside Buffer，快表/转换后备缓冲)**：
- 页表存储在内存中，每次地址转换都要访问内存查页表，速度慢
- TLB 是一个高速缓存（通常在 CPU 内部），缓存最近使用的页表项
- 命中 TLB：直接得到页框号，只需 1 次内存访问（访问目标数据）
- 未命中 TLB：需要先访问内存中的页表（1次），再访问目标数据（1次），共 2 次
- TLB 命中率通常 > 99%（局部性原理），对性能至关重要

### 2.3 分段存储 (Segmentation)

分段按照程序的**逻辑结构**来划分内存，每个段对应程序中的一个逻辑模块（代码段、数据段、栈段、堆段等）。

- 逻辑地址 = | 段号 S | 段内偏移 W |
- 段表：每项包含段基址（段在物理内存的起始地址）和段长度（用于越界检查）
- 地址转换：查段表得基址 B → 检查 W < 段长 → 物理地址 = B + W

**分页 vs 分段**：

| 对比 | 分页 | 分段 |
|------|------|------|
| 划分依据 | 物理（固定大小） | 逻辑（按程序结构） |
| 碎片 | 内部碎片（最后一页） | 外部碎片 |
| 用户可见 | 对用户透明 | 用户可见（段有语义） |
| 共享和保护 | 困难（页无语义） | 方便（按逻辑段共享/保护） |

**段页式**：先按逻辑分段，再在每段内分页。逻辑地址 = | 段号 | 页号 | 页内偏移 |。结合两者优点，是许多实际系统的选择。

### 2.4 虚拟内存 (Virtual Memory)

**核心思想**：让程序认为它拥有一大片连续的内存空间（虚拟地址空间），实际上只有正在使用的部分在物理内存中，其余在磁盘上。运行时按需调入。

**理论基础——局部性原理**：
- **时间局部性**：刚被访问的数据近期可能再次被访问（如循环变量）
- **空间局部性**：刚被访问的数据附近的数据可能很快被访问（如数组遍历）
- 因此，程序在任何时刻只需要一小部分页面在内存中即可正常运行

**请求分页 (Demand Paging)**：页面不预先全部加载到内存，而是在首次访问时才调入。在基本分页基础上增加以下功能：
- 页表项的 Valid 位初始为 0（页面不在内存）
- CPU 访问 Valid=0 的页 → 触发 **Page Fault（缺页中断）**
- OS 处理 Page Fault：从磁盘调入该页到空闲页框，设 Valid=1，重新执行触发异常的指令
- 如果没有空闲页框，使用页面置换算法选择一个页换出

**Demand Paging 的优势**：进程启动时无需加载全部页面（甚至可以零页面启动），大幅减少启动时间和内存占用。依赖局部性原理，实际运行中大部分访问集中在少量页面。

**Copy-on-Write (COW，写时复制)**：

fork() 创建子进程时的优化技术：
1. fork() 时**不复制**父进程的物理页面，父子进程共享同一组物理页
2. 将所有共享页的 PTE 标记为**只读（Write=0）**
3. 当任一进程试图**写入**共享页时，触发 Page Fault（写保护异常）
4. OS 在异常处理中：分配新物理页 → 复制原页内容 → 更新触发异常的进程的 PTE 指向新页并设为可写 → 重新执行写指令
5. 对于从不写入的页（如代码段），始终共享，永远不会复制

**效果**：fork() + exec() 的常见模式中，exec() 会替换整个地址空间，COW 避免了复制后立即丢弃的浪费。

**Zero Fill on Demand（按需零填充）**：

当进程请求新的内存页（如堆扩展、栈增长）时：
1. OS 并不立即分配物理页，只在页表中标记 Valid=0
2. 进程首次访问该页时触发 Page Fault
3. OS 分配一个物理页框，**填充全零**，映射到进程地址空间
4. 安全性：防止进程读到前一个使用者残留的数据

**页面置换算法**：

| 算法 | 思想 | 优缺点 |
|------|------|--------|
| **OPT** | 淘汰未来最长时间不会被使用的页 | 理论最优，但需要知道未来的访问序列，不可实现。作为基准。 |
| **FIFO** | 淘汰最先进入内存的页 | 简单，但可能淘汰常用页。会出现 **Belady 异常**。 |
| **LRU** | 淘汰最近最久未使用的页 | 性能接近 OPT，但需要硬件支持（时间戳或栈），实现开销大。 |
| **Clock (时钟)** | LRU 的近似。给每页一个访问位，指针循环扫描：访问位=1则清零跳过，访问位=0则淘汰 | 实现简单高效，实际 OS 常用。 |
| **改进 Clock** | 同时考虑访问位 (A) 和修改位 (M)。优先淘汰 (0,0)，其次 (0,1)，以此类推 | 减少换出修改过的页（脏页写回磁盘的 I/O 开销）。 |

**Belady 异常**：FIFO 算法独有的怪象——增加物理页框数，缺页率反而上升。原因是 FIFO 不是**栈算法**（即 n 个页框包含的页面集合不是 n+1 个页框的子集）。LRU 和 OPT 是栈算法，不会出现此异常。

**抖动 (Thrashing)**：当分配给进程的物理页框过少时，进程频繁缺页，大部分时间都在进行页面换入换出（I/O），几乎无法执行有效计算。CPU 利用率反而下降。

**工作集 (Working Set)**：进程在某个时间窗口 Δ 内访问的页面集合。如果物理内存能容纳所有进程的工作集，就不会发生抖动。工作集模型是操作系统决定为进程分配多少页框的依据。

## 三、文件系统

### 3.1 文件的概念

文件是操作系统对存储设备上数据的抽象。用户通过文件名访问数据，无需关心数据在磁盘上的物理位置。

**文件逻辑结构**：
- **无结构文件（流式文件）**：文件是一系列字节流，没有内部结构。UNIX/Linux 中的大多数文件属于此类。应用程序自行解释字节的含义。
- **有结构文件（记录式文件）**：文件由固定格式的记录组成，每条记录有若干字段。如数据库的表文件。

### 3.2 目录结构

目录（文件夹）用于组织和管理文件。目录本身也是一种特殊文件，其内容是一张表，每项将文件名映射到文件的元数据（FCB 或 inode 号）。

- **单级目录**：所有文件在一个目录中，不允许重名。简单但不实用。
- **两级目录**：为每个用户设一个目录。解决不同用户间的文件重名问题。
- **树形目录**：目录可以嵌套形成树状层次结构。支持绝对路径和相对路径。现代操作系统的标准。
- **无环图目录**：允许不同目录项指向同一个文件或目录（共享），形成有向无环图。通过硬链接或符号链接实现。

### 3.3 文件存储方式

文件数据在磁盘上的物理组织方式，影响读写性能和空间利用率。

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **连续分配** | 文件占据磁盘上一段连续的块 | 顺序和随机访问都快（直接计算块号） | 外部碎片严重，文件大小不易动态增长 |
| **链接分配** | 每个磁盘块包含指向下一块的指针，形成链表 | 无外部碎片，文件可动态增长 | 只支持顺序访问（需沿链表遍历），指针占用空间 |
| **索引分配** | 为每个文件建立一个索引块，索引块中存放所有数据块的地址 | 支持随机访问，无碎片 | 索引块本身占用空间 |

**FAT（文件分配表）**：链接分配的改进。将所有块的链接指针集中存放在一张表中（FAT），整张表加载到内存，随机访问变为查表操作。

**UNIX inode 索引**：采用混合索引方式，兼顾小文件和大文件：
- 12 个**直接索引**：直接存放数据块号（覆盖 48KB）
- 1 个**一级间接索引**：指向一个索引块，该索引块存放数据块号
- 1 个**二级间接索引**：指向一个索引块，其中每项再指向一个索引块
- 1 个**三级间接索引**：再多一层间接
- 绝大多数文件是小文件，通过直接索引就能访问，效率很高

### 3.4 空闲空间管理

操作系统需要跟踪磁盘上哪些块是空闲的。

- **位图法 (Bitmap)**：用一个 bit 数组表示所有磁盘块的状态，0=空闲，1=占用。查找空闲块可利用位运算加速。空间开销小（1TB 磁盘 = 256MB 位图）。
- **空闲链表**：将所有空闲块链接成链表。分配和回收只操作链表头，但查找连续空闲块效率低。
- **空闲表**：记录每段连续空闲区域的起始地址和长度。适合连续分配。
- **成组链接法**：UNIX 采用。将空闲块分组，每组的第一个块记录下一组的信息。结合了链表和表方法的优点。

### 3.5 磁盘调度算法

磁盘访问时间 = 寻道时间（移动磁头到目标柱面）+ 旋转延迟 + 传输时间。寻道时间是主要瓶颈，磁盘调度的目标是最小化总寻道距离。

| 算法 | 描述 | 特点 |
|------|------|------|
| **FCFS** | 按请求到达的顺序服务 | 公平但寻道距离可能很大 |
| **SSTF** | 每次选择距当前磁头最近的请求 | 性能好但可能饥饿（远处请求一直不被服务） |
| **SCAN（电梯算法）** | 磁头从一端向另一端扫描，途中服务请求，到达端点后反向 | 消除饥饿，但刚扫过的区域等待时间长 |
| **C-SCAN** | 单方向扫描，到达一端后快速回到另一端重新开始 | 等待时间更均匀 |
| **LOOK / C-LOOK** | 类似 SCAN/C-SCAN，但不必到达最边缘，到最远请求即可反向/回头 | 实际操作系统中最常用 |

## 四、I/O 管理

### 4.1 I/O 设备通信方式

**CPU 如何与设备交互**：

| 方式 | 原理 | 特点 |
|------|------|------|
| **Port-Mapped I/O** | 使用专用的 I/O 指令（如 x86 的 in/out）访问设备寄存器，设备有独立的 I/O 地址空间 | 地址空间隔离，但指令有限 |
| **Memory-Mapped I/O** | 将设备寄存器映射到物理内存地址空间，用普通 load/store 指令访问 | 编程方便（可用 C 指针操作），现代系统主流 |

**I/O 通知方式**：

| 方式 | 原理 | 适用场景 |
|------|------|----------|
| **中断 (Interrupt)** | 设备完成操作后主动通知 CPU | 低频 I/O，CPU 不浪费时间等待 |
| **轮询 (Polling)** | CPU 主动循环查询设备状态寄存器 | 高频 I/O（如高速网卡），避免中断开销 |
| **混合方式** | 高频时轮询，低频时中断 | 实际系统常用（如 Linux NAPI） |

### 4.2 I/O 控制方式

I/O 控制方式决定了 CPU 与 I/O 设备之间如何协调数据传输。从早期到现代，经历了四种方式的演进：

| 方式 | 原理 | CPU 利用率 | 特点 |
|------|------|-----------|------|
| **程序直接控制（轮询）** | CPU 发出 I/O 命令后不断查询设备状态，直到完成 | 极低（CPU 忙等） | 最简单，CPU 一直被 I/O 占用 |
| **中断驱动** | CPU 发出 I/O 命令后去做别的事，设备完成后发中断通知 CPU | 较高 | 每传输一个字节/字都要中断一次，频繁中断有开销 |
| **DMA (直接存储器访问)** | DMA 控制器负责在设备和内存之间批量传输数据，完成后才中断 CPU | 高 | CPU 只在开始和结束时介入，适合块设备 |
| **通道** | 独立的 I/O 处理器，执行通道程序，可以管理多个设备 | 最高 | 大型机采用，CPU 完全从 I/O 中解放 |

### 4.3 I/O 接口类型（用户视角）

| 接口类型 | 行为 | 适用场景 |
|----------|------|----------|
| **阻塞 (Blocking)** | 进程发起 I/O 后被挂起，直到操作完成后才返回 | 最常见的默认行为，编程简单 |
| **非阻塞 (Non-blocking)** | I/O 调用立即返回（可能返回部分结果或"未就绪"状态），进程可继续执行 | 需要轮询检查完成状态 |
| **异步 (Asynchronous)** | I/O 调用立即返回，操作完成后通过回调/信号/事件通知进程 | 高并发服务器（如 epoll/io_uring） |

### 4.4 存储设备

#### 磁盘 (HDD)

**物理结构**：盘片（Platter）、磁道（Track）、扇区（Sector，通常 512B）、柱面（Cylinder）、磁头（Head）。

**访问时间** = 寻道时间（Seek Time）+ 旋转延迟（Rotational Latency）+ 传输时间（Transfer Time）
- 寻道时间：磁头移动到目标柱面，最慢（毫秒级）
- 旋转延迟：等待目标扇区旋转到磁头下方，平均为半圈时间（7200 RPM ≈ 4.2ms）
- 传输时间：实际读写数据，最快

#### 固态硬盘 (SSD)

**基本结构**：NAND Flash 芯片 + 控制器（含 FTL）

**NAND Flash 特性**：
- 读取单位：Page（通常 4KB-16KB）
- 写入单位：Page（必须写入已擦除的空白 Page）
- 擦除单位：Block（包含多个 Page，通常 256KB-数 MB）
- **不能原地更新**：写入前必须先擦除整个 Block

**FTL (Flash Translation Layer)**：SSD 的核心固件层，功能类似虚拟内存的"间接层"：
- **地址映射**：逻辑块号 (LBA) → 物理 Page 地址，对 OS 提供与 HDD 兼容的块设备接口
- **写入方式 (Copy-on-Write)**：修改数据时不覆盖原 Page，而是写入新的空白 Page，旧 Page 标记为无效
- **垃圾回收 (GC)**：回收包含大量无效 Page 的 Block（将有效 Page 搬移到其他 Block 后擦除）
- **磨损均衡 (Wear Leveling)**：均匀分配擦写次数到各 Block（NAND 有擦写寿命限制）

**写放大 (Write Amplification)**：实际写入 Flash 的数据量 / 用户请求写入的数据量。因垃圾回收需要搬移有效数据，写放大 > 1，影响 SSD 寿命和性能。

**SSD vs HDD**：

| 特性 | HDD | SSD |
|------|-----|-----|
| 随机读写 | 慢（需寻道+旋转） | 快（无机械部件） |
| 顺序读写 | 较快 | 非常快 |
| 功耗 | 高 | 低 |
| 抗震性 | 差（机械部件） | 好 |
| 寿命 | 理论无限次读写 | 有擦写次数限制 |
| 价格/容量 | 低 | 高 |

### 4.5 I/O 性能

**延迟 (Latency)**：完成一次 I/O 操作的总时间。
**吞吐量 (Throughput/Bandwidth)**：单位时间内完成的 I/O 操作数或数据量。

两者经常矛盾：批量处理可以提高吞吐量，但增加单个请求的排队延迟。

**排队理论基础**：

当 I/O 请求排队等待服务时：
- **到达率 λ**：单位时间到达的请求数
- **服务率 μ**：单位时间能处理的请求数
- **利用率 U = λ / μ**：设备忙碌的时间比例（必须 U < 1 系统才稳定）
- 当 U → 1 时，排队延迟急剧增长（趋向无穷），因此不能让设备持续满载运行
- 实际系统通常将利用率控制在 70%-80% 以下以保证合理的响应时间

### 4.6 缓冲技术

缓冲区是内存中的一块区域，用于缓和 CPU 与 I/O 设备速度不匹配的问题。

- **单缓冲**：一个缓冲区在 CPU 处理数据和 I/O 设备传输数据之间交替使用
- **双缓冲**：两个缓冲区，CPU 和 I/O 设备可以同时工作（一个在传输，一个在处理）
- **循环缓冲**：多个缓冲区组成环形，进一步提高并行度
- **缓冲池**：由系统统一管理的公用缓冲区集合

### 4.7 SPOOLing（假脱机技术）

SPOOLing (Simultaneous Peripheral Operations On-Line) 将独占设备改造为共享设备的技术。在磁盘上开辟输入/输出井（模拟输入/输出设备），所有进程的 I/O 请求先排队到磁盘上的缓冲区，由 SPOOLing 程序统一管理。

最典型的例子是**打印机共享**：多个进程的打印任务先写入磁盘上的打印队列，打印守护进程按顺序将它们发送到打印机。

## 五、进程间通信 (IPC)

进程间通信 (Inter-Process Communication) 是不同进程之间交换数据和信息的机制。由于进程有独立的地址空间，不能直接访问彼此的内存，需要操作系统提供专门的通信机制。

| 方式 | 原理 | 特点 | 适用场景 |
|------|------|------|----------|
| **管道 (Pipe)** | 内核中的一块缓冲区，一端写入一端读出，数据按字节流传输 | 半双工（单向），仅限父子进程之间 | 简单的单向数据传输，如 shell 中的 `cmd1 \| cmd2` |
| **命名管道 (FIFO)** | 与管道类似，但有文件系统中的名字，任意进程可通过该名字访问 | 半双工，不限于父子进程 | 无亲缘关系进程间的简单通信 |
| **消息队列** | 内核维护的消息链表，进程以格式化的消息（类型+数据）收发 | 异步，有类型标识可以选择性接收 | 不同类型消息的异步通信 |
| **共享内存** | 多个进程映射同一块物理内存到各自的地址空间，直接读写 | **最快**的 IPC（无需内核介入），但需配合信号量等同步机制 | 大量数据的高速交换 |
| **信号量** | 用于进程间的同步与互斥控制，不传输实际数据 | 只传递"同步信号" | 进程同步，保护临界区 |
| **信号 (Signal)** | 软件层面的中断，异步通知进程某个事件发生 | 不传输数据，只传递事件类型编号 | 异常处理（SIGKILL杀进程、SIGALRM定时器等） |
| **Socket** | 通用的通信接口，支持同一机器和不同机器间的通信 | 支持 TCP/UDP，最通用但开销较大 | 网络通信、分布式系统 |

## 六、用户态与内核态

**为什么需要区分？** 操作系统需要保护自身和硬件不被用户程序破坏。CPU 通过**特权级别**机制来实现这种保护。

- **用户态 (User Mode)**：程序只能执行非特权指令，只能访问自己的地址空间。不能直接操作硬件、修改页表、关中断等。绝大多数应用程序在用户态运行。
- **内核态 (Kernel Mode)**：操作系统内核运行的模式，可以执行所有指令（包括特权指令），可以访问所有内存和硬件设备。

**切换时机**（用户态 → 内核态）：
1. **系统调用 (System Call)**：用户程序主动请求操作系统服务（如 read、write、fork）
2. **中断 (Interrupt)**：外部设备发出中断请求（如键盘输入、磁盘完成）
3. **异常 (Exception)**：程序执行中出现错误（如除零、越界、缺页）

**系统调用过程**：
1. 用户程序将系统调用号和参数放入寄存器
2. 执行**陷入指令 (trap/syscall/int 0x80)** → CPU 切换到内核态
3. 内核根据系统调用号查找系统调用表，执行对应的内核函数
4. 执行完毕，将结果放入寄存器，返回用户态

系统调用是用户程序与内核之间的唯一合法接口。
